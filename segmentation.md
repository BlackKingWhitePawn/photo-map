Это не только кластеризация, а сегментация истории перемещений

Ваше условие описывает поиск непрерывного временного эпизода:

обычная география
    -> переход в другой устойчивый регион
        -> нахождение там с даты A по дату B
    -> возврат к обычной географии

Поэтому обычного DBSCAN по трём координатам (latitude, longitude, time) недостаточно. Нужна комбинация:

Сегментации временного ряда - найти границы A и B.
Пространственной кластеризации - определить места внутри этого интервала.
Классификации сегмента - решить, является ли найденный интервал поездкой.

В существующей модели у вас уже есть необходимые поля координат и takenAt, а автоматическое объединение в поездки ранее было отделено от визуальной кластеризации MapLibre. Это правильная архитектура: «поездки» должны быть отдельным семантическим слоем над фотографиями.

Какие алгоритмы существуют
Метод	Что делает	Насколько подходит
ST-DBSCAN	Одновременно учитывает пространственный и временной радиусы	Подходит для отдельных событий и остановок
Stay Point / Stay Region Detection	Ищет интервалы длительного нахождения в ограниченной области	Хорошая основа, но фотографии слишком разрежены по сравнению с GPS
HDBSCAN / OPTICS с временной метрикой	Находит пространственно-временные группы разной плотности	Полезен для мест внутри поездки
Change Point Detection	Ищет моменты смены географического режима	Лучше всего находит даты A и B
HMM / HSMM	Определяет состояния «дома», «в пути», «в поездке»	Гибко, но сложно реализовать и настраивать
Динамическая сегментация	Делит временную последовательность на непрерывные географически однородные части	Наиболее точно соответствует вашей формулировке
1. ST-DBSCAN

ST-DBSCAN расширяет DBSCAN двумя независимыми условиями соседства:

пространственное расстояние <= epsSpace
временное расстояние <= epsTime

Например:

epsSpace = 30 км
epsTime = 3 дня
minPoints = 5

Фотографии объединяются, только когда они близки и по месту, и по времени. Алгоритм предназначен именно для пространственно-временных данных и умеет отделять шум.

Преимущества
относительно просто реализовать;
не нужно задавать количество поездок;
повторные поездки в одно место могут стать разными кластерами;
отдельные случайные фотографии можно признать шумом.
Проблемы
нужен один фиксированный пространственный радиус;
поездка по нескольким городам распадётся на несколько кластеров;
длинная поездка может разделиться из-за дней без фотографий;
цепочка промежуточных точек может соединить слишком далёкие места;
алгоритм не моделирует явно «уехал из дома - вернулся домой».

Вывод: ST-DBSCAN подходит для поиска отдельных локаций или событий внутри поездки, но не лучший основной детектор поездок.

2. Stay Point Detection

Классический алгоритм ищет последовательность точек, которые в течение некоторого времени остаются внутри радиуса:

distance <= D
duration >= T

Такие алгоритмы широко применяются для выделения значимых мест из GPS-траекторий. В классических экспериментах встречались параметры вроде 200 метров и 20 минут, но для фотографий масштаб должен быть гораздо больше.

Адаптация для фотографий:

радиус = 10-50 км
продолжительность = 1-30 дней
минимум активных дней = 1-3
Проблема

GPS-трек может содержать точку каждые несколько секунд, а фотографии могут отсутствовать целые сутки. Поэтому отсутствие фотографии нельзя считать доказательством отсутствия пользователя в этом месте.

Вывод: концептуально подходит, но нужен вариант stay region, устойчивый к большим временным пробелам.

3. HDBSCAN с пространственно-временной метрикой

Можно определить собственное расстояние между фотографиями:

space = haversineDistance(photoA, photoB) / spatialScale
time = abs(timeA - timeB) / temporalScale

combinedDistance = max(space, time)

Например:

spatialScale = 30 км
temporalScale = 3 дня

HDBSCAN строит иерархию плотностных групп и устойчивее обычного DBSCAN при различной плотности данных. Он также умеет оставлять точки шумом и назначать степень принадлежности к кластеру.

Недостаток

HDBSCAN не гарантирует, что кластер является одним непрерывным временным интервалом. При неудачной метрике он может:

объединить две поездки в один город;
разделить одну поездку на несколько групп;
смешать понятия «поездка» и «место внутри поездки».

Вывод: хорошо использовать после определения границ поездки, чтобы найти посещённые города и места.

4. Change Point Detection

Это наиболее близкий класс алгоритмов к вашему описанию.

Вы строите временной ряд:

день 1 -> Екатеринбург
день 2 -> Екатеринбург
день 3 -> Казань
день 4 -> Казань
день 5 -> Казань
день 6 -> Екатеринбург

Алгоритм ищет точки изменения:

A = переход Екатеринбург -> Казань
B = переход Казань -> Екатеринбург

Для пакетной обработки всей галереи хорошо подходит PELT - алгоритм поиска нескольких точек изменения с функцией стоимости и штрафом за чрезмерное число сегментов.

Для обработки новых фотографий по мере их появления существует Bayesian Online Change Point Detection, который оценивает вероятность начала нового режима онлайн.

Что подавать в алгоритм

Не отдельные фотографии, а признаки каждого дня или фотосессии:

centerLatitude
centerLongitude
distanceToHome
spatialDispersion
dominantH3Cell
numberOfPhotos

PELT разделит последовательность на периоды с относительно стабильными географическими характеристиками.

Вывод: лучший способ найти даты начала и окончания поездки.

5. HMM или HSMM

Можно представить историю как последовательность скрытых состояний:

HOME
AWAY
TRANSIT
UNKNOWN

Алгоритм получает наблюдения:

расстояние от домашнего региона;
пространственный разброс;
число фотографий;
время с предыдущей фотографии;
принадлежность основному географическому кластеру.

HMM оценивает наиболее вероятную последовательность состояний:

HOME -> TRANSIT -> AWAY -> TRANSIT -> HOME

HSMM дополнительно моделирует ожидаемую длительность каждого состояния.

Преимущества
естественно работает с неопределённостью;
можно учитывать дни без фотографий;
хорошо разделяет пребывание и дорогу;
может присваивать вероятность поездки.
Недостатки
сложнее тестировать;
нужно настраивать вероятности переходов;
для первого релиза избыточен;
без размеченных примеров параметры будут эвристическими.
Лучший вариант для вашего приложения

Я рекомендую гибрид:

временная агрегация
+
определение базовых мест
+
PELT или динамическая сегментация
+
пространственный HDBSCAN внутри сегментов
+
правила подтверждения поездки
Почему не один ST-DBSCAN

Потому что поездка является интервалом отсутствия в обычном регионе, а не обязательно одной плотной географической группой.

Например:

1 июля - Екатеринбург
2 июля - Москва
4 июля - Санкт-Петербург
7 июля - Псков
9 июля - Екатеринбург

Это одна поездка, но как минимум три пространственных кластера.

Поэтому нужно разделить понятия:

Trip - временной интервал поездки.
Destination - город или локальное место внутри поездки.
Photo cluster - группа близких фотографий.
Transit photos - фотографии в пути.
Предлагаемый алгоритм
Этап 1. Подготовить временные наблюдения

Сначала отсортировать фотографии:

photos.sortedBy { it.takenAt }

Затем объединить серийные фотографии в фотосессии.

Пример правила:

разрыв по времени <= 3 часа
и
расстояние <= 20 км

Двадцать фотографий, сделанных за пять минут, должны давать примерно такой же вес, как одна фотосессия, а не в 20 раз больший.

Особенно важно считать доли по активным дням или фотосессиям, а не только по количеству фотографий:

500 фотографий за один день
не должны перевешивать
по одной фотографии в течение десяти дней
Этап 2. Определить базовые регионы

Нужно определить обычные места пользователя:

основной город;
дом или домашний район;
второй регулярно посещаемый город;
работа или университет, если требуется такая детализация.

Для этого можно кластеризовать все фотосессии пространственно через HDBSCAN или OPTICS, а вес региона считать по количеству уникальных дней посещения.

baseScore =
    uniqueVisitedDays
    + activeMonths * monthWeight

Главный базовый регион - не обязательно место с максимальным числом фотографий. Лучше выбирать место, встречающееся в наибольшем числе разных дней и месяцев.

Можно поддерживать несколько баз:

Екатеринбург - основная база
Москва - частая база
родной город - дополнительная база

Иначе каждое регулярное возвращение во второй город будет ошибочно определяться как поездка.

Этап 3. Построить дневной временной ряд

Для каждого дня:

data class DailyLocationObservation(
    val date: LocalDate,
    val centerLatitude: Double?,
    val centerLongitude: Double?,
    val distanceToNearestBaseKm: Double?,
    val dispersionKm: Double?,
    val dominantRegionId: String?,
    val sessionCount: Int
)

Центр дня лучше считать географической медианой или медоидом, а не обычным средним.

Этап 4. Найти точки изменения

PELT или собственный динамический алгоритм разделяет историю на непрерывные сегменты:

Segment 1: Екатеринбург
Segment 2: Казань
Segment 3: Екатеринбург

Функция стоимости сегмента может учитывать:

пространственный разброс
+
расстояние от единого центра
+
смену доминирующей H3-ячейки
+
расстояние до базового региона

Упрощённо:

segmentCost =
    sum(robustDistance(point, segmentMedoid))
    + outlierPenalty
    + missingDayPenalty

Алгоритм ищет разбиение:

минимум:
сумма стоимости сегментов
+
penalty * количество сегментов

Штраф не позволяет создавать отдельный сегмент для каждого дня.

Этап 5. Проверить, является ли сегмент поездкой

Для сегмента [A, B] вычислить:

duringShare
    доля дней или сессий внутри региона поездки

boundaryContrast
    насколько география до A и после B отличается от региона поездки

distanceFromBase
    расстояние центра поездки до ближайшей базы

activeDays
    количество дней с фотографиями

duration
    календарная длительность

dispersion
    пространственный разброс

Пример условий:

distanceFromBase >= 50 км

duringShare >= 0.65

boundaryContrast >= 0.70

activeDays >= 1

duration <= 60 дней

Это стартовые значения, а не универсальные константы.

Этап 6. Определить центр и радиус

Для одноцентровой поездки:

центр = географический медоид фотосессий
радиус = 80-й процентиль расстояний до центра

Почему не максимальное расстояние:

20 фотографий находятся в Казани
1 фотография снята из самолёта в 400 км

Максимум сделал бы радиус бессмысленным. Процентиль сохранит устойчивость к выбросам.

R = clamp(Q80(distanceToCenter), minimumRadius, maximumRadius)

Например:

minimumRadius = 2 км
maximumRadius = 50 км
Этап 7. Найти места внутри поездки

После подтверждения интервала [A, B] запустить пространственный HDBSCAN или DBSCAN только по фотографиям этого интервала.

Результат:

Поездка: 10-17 июля

Места:
- Казань
- Свияжск
- Голубые озёра

Шум:
- фотография из самолёта
- фотография на трассе
Формализация вашего правила

Пусть:

P[A,B] - фотосессии между A и B
C - медоид этих фотосессий
R - устойчивый радиус

Тогда:

insideDuring =
    количество активных дней внутри Circle(C, R)
    /
    количество активных дней между A и B

Контекст до и после:

outsideBefore =
    доля активных дней до A вне Circle(C, R)

outsideAfter =
    доля активных дней после B вне Circle(C, R)

Кандидат является поездкой, если:

insideDuring >= 0.65

outsideBefore >= 0.70

outsideAfter >= 0.70

distance(C, nearestBase) >= 50 км

Полезно рассчитывать итоговую уверенность:

tripConfidence =
    0.35 * insideDuring
    + 0.25 * min(outsideBefore, outsideAfter)
    + 0.20 * distanceFromBaseScore
    + 0.10 * temporalContinuity
    + 0.10 * activeDayCoverage
Одноцентровые и многоцентровые поездки

Не стоит требовать, чтобы любая поездка укладывалась в один круг R.

Одноцентровая
Екатеринбург -> Казань -> Екатеринбург

Большинство фотографий находится около Казани. Здесь ваша модель работает напрямую.

Многоцентровая
Екатеринбург
-> Москва
-> Санкт-Петербург
-> Выборг
-> Екатеринбург

Большинство фотографий может не помещаться в разумный радиус. Но временной интервал явно является одной поездкой.

Поэтому классификация должна быть такой:

если один пространственный кластер покрывает >= 65% сессий:
    SINGLE_DESTINATION_TRIP

иначе если несколько удалённых от базы кластеров
идут подряд во времени:
    MULTI_DESTINATION_TRIP
Обработка дней без фотографий

День без фотографий не означает возвращение домой.

Рекомендуемые правила:

пробел 1-2 дня внутри устойчивого away-сегмента не разрывает поездку;
пробел больше 2 дней обозначается как UNKNOWN;
поездка продолжается, если после пробела фотографии снова появляются в том же регионе;
окончание лучше подтверждать фотографиями из базового региона;
если подтверждения нет, снизить tripConfidence, а не обязательно удалять поездку.
Практический выбор для первой версии
MVP без сложного машинного обучения

Используйте:

1. Агрегация фотографий в сессии.
2. Определение базового региона через HDBSCAN.
3. Дневная последовательность географических регионов.
4. Динамическая сегментация или PELT.
5. Проверка расстояния от базы и контраста границ.
6. DBSCAN/HDBSCAN для мест внутри поездки.
7. Ручное подтверждение и редактирование результата.
Итоговая рекомендация

Самый подходящий основной алгоритм для вашей задачи:

PELT или собственная динамическая сегментация непрерывной временной последовательности, дополненная HDBSCAN для пространственных мест внутри каждого временного сегмента.

ST-DBSCAN можно использовать как более простой прототип, но он хуже отражает саму сущность поездки и особенно плохо подходит для маршрутов через несколько городов. Алгоритмы с временными ограничениями действительно рассматриваются как отдельный класс методов анализа человеческой мобильности, поскольку чисто пространственная кластеризация не учитывает порядок и продолжительность посещений
