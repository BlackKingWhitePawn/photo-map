# Правила релизов

## Цель

Релиз фиксирует проверенное состояние приложения, которое можно установить на устройство или эмулятор и передать для ручной проверки.

До готовности MVP релизы имеют статус предварительных и используют версии `0.x.y`.

## Версионирование

Использовать SemVer:

```text
MAJOR.MINOR.PATCH
```

Правила:

* `MAJOR` увеличивать при несовместимых изменениях архитектуры, пользовательских данных или сценария обновления;
* `MINOR` увеличивать при добавлении новых пользовательских возможностей;
* `PATCH` увеличивать при исправлениях ошибок без добавления новых возможностей;
* до первого MVP использовать версии `0.x.y`;
* первая базовая версия после инициализации проекта - `0.1.0`;
* Android `versionName` должен совпадать с SemVer;
* Android `versionCode` должен увеличиваться на 1 для каждой сборки, предназначенной для установки вне локальной разработки.

## Имена релизных веток и тегов

Релизная ветка:

```text
release-<major>-<minor>-<patch>
```

Пример:

```text
release-0-1-0
```

Релизный тег:

```text
v<major>.<minor>.<patch>
```

Пример:

```text
v0.1.0
```

## Подготовка релизной ветки

1. Убедиться, что рабочая ветка завершена и её `branch.md` соответствует фактически выполненной работе.
2. Убедиться, что в репозитории нет секретов, ключей tile-провайдера, локальных IDE-файлов и артефактов сборки.
3. Обновить `versionName` и `versionCode` в Gradle-конфигурации.
4. Обновить README, если изменились запуск, ограничения, разрешения, настройка карты или пользовательские возможности.
5. Обновить `CHANGELOG.md`.
6. Создать релизную ветку от актуальной `main`.
7. Внести только релизные правки: версии, документацию, release notes и мелкие исправления блокирующих ошибок.

## Проверки перед релизом

Перед созданием тега релиз должен пройти:

* сборку debug-версии;
* unit-тесты;
* instrumented-тесты, если они уже добавлены в проект;
* ручной запуск приложения на устройстве или эмуляторе;
* проверку отсутствия секретов и ключей tile-провайдера в Git;
* проверку базовых сценариев, входящих в текущую версию.

Команды проекта:

```bash
./gradlew assembleDebug
./gradlew test
./gradlew connectedAndroidTest
```

Если часть проверок невозможно выполнить, это нужно явно указать в описании релиза.

## Хранение APK и AAB

Готовые APK и AAB не хранить в Git и не добавлять в обычные коммиты.

Правильная схема:

* исходный код, Gradle-конфигурация, README, `release.md` и release notes хранятся в Git;
* постоянные APK-релизы публикуются как assets в GitHub Releases;
* каждый release asset привязывается к конкретному Git-тегу версии;
* временные сборки для проверки pull request или commit можно хранить как CI artifacts;
* CI artifacts не считать постоянными релизами.

Не добавлять в репозиторий:

```text
app/build/outputs/apk/
app/build/outputs/bundle/
*.apk
*.aab
*.apks
```

Debug APK не использовать как пользовательский релиз. Он подходит только для локальной проверки.

Для ручного распространения использовать подписанный release APK:

```text
app/build/outputs/apk/release/app-release.apk
```

Перед публикацией переименовывать файл по версии:

```text
photomap-v0.1.0.apk
```

Для Google Play дополнительно готовить AAB:

```text
photomap-v0.1.0.aab
```

APK нужен для ручной установки и скачивания с GitHub Releases. AAB нужен для публикации через Google Play и напрямую на устройство обычно не устанавливается.

## Подпись релизов

Пользовательские релизы должны быть подписаны release-ключом.

Не хранить в Git:

```text
*.jks
*.keystore
keystore.properties
local.properties
```

Keystore хранить отдельно от репозитория и делать резервную копию. Потеря release-ключа может заблокировать обычное обновление уже установленного приложения.

Пароли подписи не записывать в `build.gradle.kts`. Использовать локальный `keystore.properties`, который исключён через `.gitignore`, или секреты CI.

## Release notes

Для каждого релиза `release-notes/v<version>.md` использовать только такой порядок и набор секций:

```text
# Photo Map v<version>

Date: YYYY-MM-DD

## Added

## Changed

## Fixed

## Version
```

APK metadata, package name, `versionCode`, `versionName`, min SDK, target SDK, permission list, APK path, size, SHA-256 и signature status записывать внутри секции `Version`, без отдельных секций `Checks`, `APK`, `Permissions` или `Known Limitations`.

`CHANGELOG.md` должен хранить краткую историю изменений по версиям. Release notes для GitHub Release можно готовить на основе соответствующего раздела `CHANGELOG.md`.

## Создание релиза

1. Закоммитить все релизные изменения.
2. Убедиться, что рабочее дерево чистое.
3. Создать тег в формате `v<version>`.
4. Собрать подписанный release APK.
5. Переименовать APK в формате `photomap-v<version>.apk`.
6. Создать GitHub Release для тега.
7. Прикрепить APK к GitHub Release как asset.
8. Указать в описании тега или GitHub Release release notes из этого документа.
9. После проверки релиза слить релизную ветку в `main`.

Одна версия должна соответствовать одному commit, одному тегу и одному неизменяемому APK. Не заменять файл внутри уже опубликованного релиза новой сборкой с тем же номером версии. Для исправлений выпускать новый `PATCH`-релиз.

## Hotfix-релизы

Hotfix использовать только для исправления критичной ошибки в уже выпущенной версии.

Правила:

* ветку создавать от релизного тега или актуальной стабильной ветки;
* имя ветки: `hotfix-<number>-<short-slug>`;
* менять только код и документацию, необходимые для исправления ошибки;
* увеличивать `PATCH`;
* после выпуска переносить исправление обратно в `main`.

## Ограничения

Не считать релизом произвольную локальную сборку без версии, тега и описания изменений.

Не публиковать релиз, если в репозитории есть реальные секреты, ключи tile-провайдера, приватные данные или тяжёлые артефакты сборки.
