# Задача: добавить кластеризацию фотографий на карте MapLibre

## 1. Контекст

Ты работаешь в существующем Android-приложении локальной фотогалереи.

Приложение получает фотографии с устройства, извлекает координаты из EXIF-метаданных и отображает фотографии на карте с использованием MapLibre.

Не создавай новый проект и не переписывай существующую архитектуру.

Перед изменениями:

1. Изучи структуру проекта.
2. Найди экран карты.
3. Найди существующий источник фотографий.
4. Определи, используется Kotlin или Java.
5. Определи, используется Compose или XML.
6. Найди текущую версию MapLibre.
7. Найди существующие ViewModel, Repository и модели фотографий.
8. Проверь, как сейчас загружаются координаты EXIF.
9. Проверь существующую навигацию к экрану просмотра фотографии.
10. Составь краткий список файлов, которые потребуется изменить.

Используй существующий язык, UI-фреймворк, архитектуру, DI и соглашения именования проекта.

Не обновляй MapLibre и другие крупные зависимости без необходимости.

---

## 2. Цель

Реализовать производительное отображение большого количества фотографий с координатами на карте.

Карта должна:

* объединять близкие точки в кластеры;
* пересчитывать представление кластеров в зависимости от масштаба;
* показывать количество фотографий в кластере;
* раскрывать кластер при нажатии;
* показывать список фотографий, если кластер нельзя раскрыть дальше;
* открывать существующий экран просмотра при нажатии на одиночную фотографию;
* корректно работать с совпадающими координатами;
* не блокировать главный поток;
* не отправлять фотографии, координаты или EXIF-данные в сеть.

---

## 3. Границы задачи

В текущей задаче реализовать только визуальную кластеризацию средствами MapLibre.

Не реализовывать:

* DBSCAN;
* OPTICS;
* HDBSCAN;
* серверную кластеризацию;
* reverse geocoding;
* загрузку координат на сервер;
* создание отдельных View-маркеров для каждой точки;
* тепловую карту;
* автоматическое объединение фотографий в поездки;
* изменение или удаление EXIF;
* изменение самих файлов фотографий.

Архитектура должна позволять позже добавить отдельный режим смысловой кластеризации фотографий.

---

## 4. Источник данных

Использовать существующий репозиторий фотографий как единственный источник данных.

Не выполнять повторное полное сканирование файлов, если координаты уже загружаются и хранятся проектом.

Минимальная модель точки карты:

```kotlin
data class PhotoMapPoint(
    val photoId: String,
    val contentUri: String,
    val latitude: Double,
    val longitude: Double,
    val takenAt: Long?,
)
```

Название и типы адаптировать под существующие модели проекта.

Не передавать в GeoJSON:

* Bitmap;
* массив байтов фотографии;
* полные EXIF-метаданные;
* большие списки идентификаторов;
* оригинальное изображение.

В GeoJSON должны передаваться только данные, необходимые для карты:

* стабильный ID;
* URI или ключ для поиска фотографии в репозитории;
* дата съёмки;
* координаты.

---

## 5. Получение EXIF-координат

Если извлечение координат уже реализовано, переиспользовать существующий код.

Если оно отсутствует:

1. Получать список изображений через `MediaStore`.
2. Формировать `content://` URI через `ContentUris`.
3. Открывать изображение через `ContentResolver`.
4. Использовать AndroidX `ExifInterface`.
5. Получать координаты через `getLatLong()`.
6. Выполнять файловые операции на `Dispatchers.IO`.
7. Закрывать все потоки и файловые дескрипторы через `use`.
8. Не читать Bitmap только ради EXIF.
9. Не читать EXIF повторно при каждом перемещении карты.
10. Переиспользовать существующий кэш или индекс фотографий.

Учитывать, что `getLatLong()` возвращает:

```text
[latitude, longitude]
```

При создании GeoJSON использовать:

```text
[longitude, latitude]
```

Не считать координату `(0, 0)` автоматически ошибочной. Она является допустимой географической координатой. Пропускать фотографию только при отсутствии EXIF GPS или при невалидных числах.

Валидация:

```text
latitude конечна и находится в диапазоне [-90, 90]
longitude конечна и находится в диапазоне [-180, 180]
```

Фотографии без GPS не должны попадать на карту, но должны оставаться доступными в обычной галерее.

---

## 6. Преобразование фотографий в GeoJSON

Создать отдельный mapper или use case, который преобразует список фотографий в `FeatureCollection`.

Предлагаемое имя:

```text
PhotoMapFeatureMapper
```

Для каждой фотографии создавать `Point`:

```kotlin
Point.fromLngLat(photo.longitude, photo.latitude)
```

Добавлять свойства:

```text
photo_id
content_uri
taken_at
```

Не использовать координаты или URI в качестве единственного ID.

Не удалять дубликаты по координатам. Несколько фотографий могут быть сделаны в одной точке.

Допускается удалять только точные дубликаты одной записи, имеющие одинаковый стабильный ID.

Создание большого `FeatureCollection` выполнять вне главного потока.

Вызов методов MapLibre, требующих UI thread, выполнять на главном потоке.

---

## 7. GeoJSON Source

Добавить один источник MapLibre:

```text
photo-map-source
```

Начальные параметры:

```kotlin
GeoJsonOptions()
    .withCluster(true)
    .withMaxZoom(22)
    .withClusterRadius(50)
    .withClusterMinPoints(2)
    .withClusterMaxZoom(22)
```

Значения вынести в константы:

```text
PHOTO_CLUSTER_RADIUS = 50
PHOTO_CLUSTER_MIN_POINTS = 2
PHOTO_MAP_SOURCE_MAX_ZOOM = 22
```

Максимальный zoom не выносить в пользовательские настройки. Zoom не должен управлять видимостью точек: источник должен сохранять кластеры/миниатюры доступными на любом масштабе карты, а пользователь настраивает только радиус и минимальное количество точек.

Не пересоздавать source при каждом перемещении камеры.

Источник и слои создаются один раз после загрузки стиля. При изменении списка фотографий обновляется только содержимое через `setGeoJson()`.

Если стиль карты перезагрузился, источник и слои необходимо корректно добавить повторно.

Не добавлять один source на каждую фотографию.

---

## 8. Слои MapLibre

Создать отдельные слои.

### 8.1. Слой кластеров

ID:

```text
photo-cluster-layer
```

Тип:

```text
CircleLayer
```

Фильтр:

```text
объекты, содержащие свойство point_count
```

Размер круга зависит от `point_count`.

Рекомендуемые диапазоны:

```text
2-9 фотографий
10-49 фотографий
50-199 фотографий
200 и более фотографий
```

Использовать цвета из ресурсов или текущей темы приложения. Не дублировать цвета строками в нескольких местах.

### 8.2. Счётчик кластера

ID:

```text
photo-cluster-count-layer
```

Тип:

```text
SymbolLayer
```

Текст брать из:

```text
point_count_abbreviated
```

Текст должен располагаться по центру круга.

### 8.3. Одиночные фотографии

ID:

```text
photo-unclustered-layer
```

Использовать `SymbolLayer` с миниатюрой фотографии фиксированного экранного размера.

Фильтр:

```text
объекты, не содержащие свойство point_count
```

Одиночные точки должны отображаться как миниатюры, а не как круги.

Размер миниатюры должен быть фиксированным по ширине и высоте на экране и не должен зависеть от zoom карты.

Не регистрировать и не хранить тысячи Bitmap внутри Style. Допускается ограниченный кеш миниатюр только для видимых одиночных объектов.

Не декодировать оригинал фотографии для маркера. Использовать уменьшенную миниатюру и локальный кеш.

---

## 9. Обработка нажатия на кластер

При нажатии:

1. Получить координату нажатия.
2. Сначала выполнить `queryRenderedFeatures` по слою кластеров.
3. Получить найденный кластер.
4. Получить `GeoJsonSource`.
5. Вызвать `getClusterExpansionZoom(clusterFeature)`.
6. Сравнить полученный zoom с текущим zoom и максимальным zoom карты.

Если кластер может раскрыться:

* плавно переместить камеру к координате кластера;
* установить expansion zoom;
* не добавлять произвольное фиксированное значение к текущему zoom.

Если кластер нельзя осмысленно раскрыть:

* получить фотографии через `getClusterLeaves`;
* открыть нижнюю панель со списком фотографий.

Условие показа списка:

```text
expansionZoom не больше текущего zoom с небольшим допуском
или
expansionZoom превышает максимальный zoom карты
```

Начальный размер страницы листьев:

```text
100 фотографий
```

Если фотографий больше, предусмотреть постраничную загрузку через `limit` и `offset`.

Не запрашивать бесконечное количество элементов через условный `Infinity`.

---

## 10. Совпадающие координаты

Особый случай:

* несколько фотографий могут иметь полностью одинаковые координаты;
* увеличение масштаба не сможет визуально разделить такие точки.

Для этого случая:

1. Не изменять координаты искусственным случайным смещением.
2. Не терять фотографии.
3. Не показывать только первую фотографию.
4. На максимальном масштабе открывать список листьев кластера.
5. Если несколько одиночных объектов всё же перекрываются, открывать список всех найденных объектов.

Дополнительный spiderfy-режим не входит в текущую задачу.

---

## 11. Нажатие на одиночную точку

Если кластер под нажатием не найден:

1. Выполнить запрос по слою одиночных фотографий.
2. Использовать небольшую прямоугольную область вокруг места нажатия, а не один точный пиксель.
3. Получить все найденные объекты.

Если найден один объект:

* получить `photo_id`;
* найти фотографию через существующий репозиторий;
* открыть существующий экран просмотра.

Если найдено несколько объектов:

* открыть нижнюю панель со списком фотографий.

Не передавать Bitmap через navigation arguments.

Передавать стабильный ID или URI в формате, уже используемом проектом.

---

## 12. Нижняя панель фотографий

Использовать существующий компонент списка или галереи, если он уже есть.

Если компонента нет, создать минимальный bottom sheet.

Он должен отображать:

* количество фотографий;
* миниатюры;
* дату съёмки, если доступна;
* возможность открыть выбранную фотографию;
* состояние загрузки;
* состояние ошибки;
* постраничную загрузку для крупных кластеров.

Миниатюры загружать существующей библиотекой изображений проекта.

Не декодировать полноразмерные изображения вручную.

Не удерживать Activity или Fragment в долгоживущих объектах.

---

## 13. Фильтрация

Если в приложении есть фильтры по:

* дате;
* альбому;
* папке;
* типу файла;
* избранному;

кластеризация должна применяться к уже отфильтрованному списку.

После изменения фильтра:

1. Получить новый список точек.
2. Построить новый `FeatureCollection` вне главного потока.
3. Обновить существующий source.
4. Не пересоздавать карту и стиль.

При частых изменениях фильтра допускается debounce 150-300 мс.

Последний запрос должен отменять предыдущую незавершённую подготовку GeoJSON.

---

## 14. Состояния интерфейса

Обработать состояния:

```text
Loading
PermissionRequired
Empty
NoGeotaggedPhotos
Content
Error
```

Разница:

* `Empty` - на устройстве нет доступных фотографий;
* `NoGeotaggedPhotos` - фотографии есть, но ни одна не содержит GPS;
* `PermissionRequired` - доступ к фотографиям не предоставлен.

При отказе в разрешении приложение не должно завершаться с ошибкой.

При частичном доступе Android показывать только доступные пользователю фотографии.

Не запрашивать `MANAGE_EXTERNAL_STORAGE`.

---

## 15. Производительность

Обязательные требования:

* никакого чтения файлов на main thread;
* никакого построения большого GeoJSON на main thread;
* один кластеризованный source;
* постоянное количество MapLibre layers;
* отсутствие одного View на каждую фотографию;
* отсутствие одного Bitmap на каждую фотографию внутри MapLibre Style;
* кластеризация не должна вручную запускаться после каждого движения карты;
* source не должен обновляться только из-за изменения zoom;
* повторное открытие экрана не должно повторно читать EXIF всех фотографий, если данные уже закэшированы;
* долгие операции должны отменяться при уничтожении соответствующего lifecycle owner.

Проверить работу минимум на наборах:

```text
0 точек
1 точка
10 точек
100 точек
1 000 точек
10 000 точек
```

На 10 000 точек:

* приложение не должно получать ANR;
* карта должна оставаться интерактивной;
* масштабирование не должно создавать тысячи Android View;
* память не должна непрерывно расти при повторных перемещениях карты;
* источник и слои не должны создаваться повторно после каждого движения камеры.

---

## 16. Конфиденциальность

Все данные фотографий должны оставаться локальными.

Запрещено:

* отправлять координаты в API;
* отправлять URI в API;
* отправлять EXIF в API;
* использовать remote logging для содержимого фотографий;
* добавлять аналитику с координатами;
* добавлять reverse geocoding;
* добавлять координаты в URL тайлов или карты;
* копировать фотографии во временные публичные директории.

Картографические тайлы могут загружаться согласно текущей конфигурации MapLibre, но пользовательские координаты не должны включаться в эти запросы.

В release-сборке не логировать:

* полный content URI;
* имя файла;
* точные GPS-координаты;
* EXIF-метаданные.

---

## 17. Обработка ошибок

Не допускать падения при:

* повреждённом EXIF;
* удалённой фотографии;
* закрытом или недоступном URI;
* `SecurityException`;
* `FileNotFoundException`;
* отсутствии координат;
* `NaN` и бесконечных координатах;
* перезагрузке MapLibre Style;
* пустом результате `queryRenderedFeatures`;
* отсутствии source;
* отсутствии layer;
* удалении фотографии между построением GeoJSON и нажатием;
* отмене coroutine;
* изменении разрешений во время работы приложения.

Ошибку одной фотографии не считать ошибкой всего списка.

## 18. Архитектура

Разделить ответственность.

Рекомендуемые компоненты:

```text
PhotoRepository
    предоставляет фотографии и метаданные

PhotoMapPointMapper
    преобразует фотографию в PhotoMapPoint

PhotoMapFeatureMapper
    преобразует PhotoMapPoint в GeoJSON FeatureCollection

PhotoMapViewModel
    хранит состояние карты и фильтрации

PhotoMapLayerController
    создаёт и обновляет MapLibre source и layers

PhotoClusterClickHandler
    обрабатывает раскрытие и получение листьев
```

Названия адаптировать под существующую архитектуру.

Не помещать:

* чтение MediaStore;
* чтение EXIF;
* построение GeoJSON;
* навигацию;
* обработку permissions;

в один Activity или Fragment.

Не создавать новые абстракции, если аналогичные компоненты уже существуют.

## 19. Unit-тесты

Добавить тесты для mapper и связанной логики.

Обязательные случаи:

1. Latitude и longitude передаются в GeoJSON в правильном порядке.
2. Отсутствующие координаты пропускаются.
3. `NaN` пропускается.
4. Бесконечные координаты пропускаются.
5. Latitude меньше `-90` пропускается.
6. Latitude больше `90` пропускается.
7. Longitude меньше `-180` пропускается.
8. Longitude больше `180` пропускается.
9. Координата `(0, 0)` сохраняется.
10. Несколько фотографий с одинаковыми координатами сохраняются.
11. Одинаковый photo ID не дублируется, если репозиторий случайно вернул одну запись дважды.
12. Пустой список создаёт пустой `FeatureCollection`.
13. `taken_at = null` не вызывает ошибку.
14. URI с нестандартными символами корректно сохраняется как property.
15. Фотографии рядом с `180` и `-180` longitude не вызывают ошибку преобразования.

Логику выбора действия при клике вынести в тестируемую функцию или use case.

Проверить:

```text
кластер раскрывается -> ZoomToCluster
кластер не раскрывается -> ShowClusterPhotos
одна точка -> OpenPhoto
несколько совпавших точек -> ShowPhotos
ничего не найдено -> NoAction
```

## 20. Инструментальные и ручные проверки

Проверить следующие сценарии:

### Сценарий 1

На карте одна фотография.

Ожидается:

* одиночная миниатюра фиксированного размера;
* нажатие открывает фотографию.

### Сценарий 2

Десять фотографий расположены рядом.

Ожидается:

* на малом масштабе один кластер;
* число внутри кластера равно 10;
* при нажатии карта приближается;
* после раскрытия появляются меньшие кластеры или одиночные миниатюры.

### Сценарий 3

Сто фотографий имеют одинаковые координаты.

Ожидается:

* фотографии не теряются;
* на максимальном масштабе нажатие открывает список;
* список содержит все 100 фотографий.

### Сценарий 4

На устройстве 10 000 фотографий с GPS.

Ожидается:

* нет ANR;
* карта реагирует на жесты;
* количество слоёв остаётся постоянным;
* при движении карты не создаётся новый source.

### Сценарий 5

Нет фотографий с GPS.

Ожидается:

* отображается состояние `NoGeotaggedPhotos`;
* приложение не падает.

### Сценарий 6

Доступ к фотографиям запрещён.

Ожидается:

* отображается понятное состояние разрешения;
* приложение не падает;
* повторный запрос происходит только после действия пользователя.

### Сценарий 7

Стиль карты был изменён или перезагружен.

Ожидается:

* source и layers восстанавливаются;
* обработчики не дублируются;
* нажатие не выполняется дважды.

## 21. Критерии приёмки

Задача считается выполненной, когда:

* фотографии с GPS отображаются на карте;
* близкие точки объединяются в кластеры;
* кластер показывает корректное количество;
* кластеры перестраиваются при изменении zoom средствами MapLibre;
* нажатие раскрывает кластер;
* совпадающие координаты открывают список фотографий;
* одиночная миниатюра открывает существующий просмотр фотографии;
* фотографии без GPS не вызывают ошибку;
* данные не отправляются в сеть;
* не используется `MANAGE_EXTERNAL_STORAGE`;
* отсутствуют файловые операции на главном потоке;
* отсутствует отдельный View для каждой точки и не создаётся неограниченный Bitmap-кеш;
* добавлены unit-тесты;
* проект успешно собирается;
* существующие тесты продолжают проходить;
* функциональность обычной галереи не сломана.
