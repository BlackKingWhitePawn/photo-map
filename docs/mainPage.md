# Техническое задание: главная страница с картой, поездками и частыми местами

## 1. Контекст

Ты работаешь в существующем Android-приложении локальной фотогалереи.

Приложение:

* получает фотографии через `MediaStore`;
* извлекает дату и GPS-координаты из EXIF;
* отображает фотографии на карте через MapLibre;
* кластеризует точки на карте;
* определяет поездки по времени и координатам;
* определяет часто посещаемые места;
* хранит рассчитанные данные локально;
* не отправляет фотографии, координаты или EXIF в сеть.

Не создавай новый проект и не переписывай существующую архитектуру.

Перед реализацией:

1. Изучи структуру проекта.
2. Найди текущий стартовый экран приложения.
3. Найди существующий экран карты.
4. Найди модели фотографий, поездок и мест.
5. Найди репозитории и Room DAO.
6. Определи, используется Jetpack Compose или XML.
7. Найди существующую навигацию.
8. Найди существующий компонент карточки фотографии или поездки.
9. Найди способ загрузки миниатюр.
10. Найди контроллер или helper для MapLibre.
11. Проверь, где хранятся результаты определения поездок.
12. Проверь, где хранятся данные тепловой карты.
13. Составь список файлов, которые потребуется создать или изменить.

Используй существующие:

* язык проекта;
* архитектурный подход;
* DI;
* тему;
* typography;
* компоненты;
* библиотеку изображений;
* механизм навигации;
* соглашения именования.

---

# 2. Цель

Создать главную страницу приложения, которая показывает личную географию пользователя через три основных раздела:

1. Большая тепловая карта.
2. Блок «Ваши поездки».
3. Блок «Ваши места».

Главная страница должна позволять быстро:

* увидеть географию поездок;
* открыть случайно выбранную поездку;
* перейти ко всем поездкам;
* посмотреть поездки на отдельной карте;
* открыть часто посещаемое место;
* посмотреть фотографии места по дням и месяцам.

---

# 3. Общая структура главной страницы

Главная страница является вертикально прокручиваемым экраном.

Рекомендуемая структура:

```text
Главная
├── Тепловая карта - 60% высоты экрана
├── Ваши поездки
│   ├── заголовок
│   ├── кнопка «Все поездки»
│   └── 3 карточки поездок
├── Карта поездок - 30% высоты экрана
└── Ваши места - 30% высоты экрана
    ├── заголовок
    ├── кнопка «Все места»
    └── карточки частых мест
```

Если экран реализован через Compose, использовать `LazyColumn`.

Не помещать весь экран в обычный `Column` с большим количеством элементов, если это приводит к одновременной инициализации всех карт и карточек.

# 4. Расчёт высоты блоков

`60vh` и `30vh` интерпретировать как процент доступной высоты окна Android.

Доступная высота должна учитывать:

* status bar;
* navigation bar;
* display cutout;
* верхнюю панель приложения;
* нижнюю навигацию, если она есть.

Рекомендуемые размеры:

```text
Тепловая карта:
60% доступной высоты экрана

Карта поездок:
30% доступной высоты экрана

Блок «Ваши места»:
минимум 30% доступной высоты экрана
```

Не вычислять высоту через случайную константу в `dp`.

Создать отдельную функцию или helper:

```kotlin
fun calculateHomeSectionHeights(
    availableHeight: Dp
): HomeSectionHeights
```

Пример модели:

```kotlin
data class HomeSectionHeights(
    val heatmapHeight: Dp,
    val tripsMapHeight: Dp,
    val placesHeight: Dp
)
```

Для небольших экранов добавить минимальные ограничения:

```text
тепловая карта - не меньше 320 dp;
карта поездок - не меньше 200 dp;
блок мест - не меньше 240 dp.
```

# 6. Верхний блок: тепловая карта

## 6.1. Назначение

Верхний блок отображает тепловую карту географии пользователя.

По умолчанию тепловая карта должна показывать активность поездок:

* число поездок;
* число активных дней;
* частоту посещения регионов;
* места назначения поездок.

Не использовать простое количество фотографий как единственный вес. Большая серия кадров за один день не должна полностью подавлять несколько отдельных поездок.

## 6.2. Размер

Высота:

```text
60% доступной высоты экрана
```

Ширина:

```text
100% доступной ширины
```

Карта может занимать всю ширину без внешнего горизонтального отступа.

Если дизайн приложения использует карточное оформление, допускается:

* горизонтальный отступ 12-16 dp;
* скругление 16-24 dp;
* обрезка содержимого по форме контейнера.

## 6.3. Содержимое карты

Использовать существующий `HeatmapLayer` MapLibre.

Источник карты должен получать уже рассчитанные данные из локального хранилища.

Главная страница не должна:

* определять поездки;
* запускать пространственную кластеризацию;
* пересчитывать H3-ячейки;
* читать EXIF;
* сканировать MediaStore;
* строить тепловую карту на main thread.

Поток:

```text
Room
-> Repository
-> ViewModel
-> готовый FeatureCollection
-> GeoJsonSource
-> HeatmapLayer
```

## 6.4. Взаимодействие

Карта должна поддерживать:

* перемещение;
* zoom;
* стандартные жесты MapLibre;
* нажатие на область высокой активности.

При нажатии на область можно:

1. Найти ближайшую heat-ячейку.
2. Получить поездки, которые внесли вклад в эту ячейку.
3. Открыть список поездок в bottom sheet.

Если такой механизм отсутствует, в первой версии достаточно открыть экран «Все поездки» с географическим фильтром по выбранной области.

## 6.5. Элементы поверх карты

Добавить компактный заголовок поверх карты:

```text
Ваша география
```

Подзаголовок:

```text
Поездки и посещённые места
```

Допускается кнопка:

```text
Открыть карту
```

Кнопка открывает существующий полноэкранный экран карты.

Текст должен иметь читаемый фон или градиент. Не размещать белый текст непосредственно поверх светлой карты без подложки.

## 6.6. Состояния

### Loading

Показывать placeholder или skeleton того же размера.

Не показывать пустой серый прямоугольник.

### Empty

Если поездок пока нет:

```text
Поездки пока не найдены
Добавьте фотографии с датой и геопозицией
```

Можно показать обычную карту без heat-слоя.

### PermissionRequired

Если нет доступа к фотографиям:

```text
Нужен доступ к фотографиям
```

Добавить кнопку запроса разрешения.

### Error

Показать карту без heat-слоя и ненавязчивое сообщение:

```text
Не удалось загрузить тепловую карту
```

Не блокировать остальную главную страницу.

# 7. Блок «Ваши поездки»

## 7.1. Заголовок

После тепловой карты добавить строку:

```text
Ваши поездки                    Все поездки
```

`Все поездки` должно быть кнопкой или текстовым действием.

При нажатии открывается экран:

```text
AllTripsScreen
```

или существующий аналог.

## 7.2. Отступы

Рекомендуемые значения:

```text
верхний отступ от карты - 24 dp;
горизонтальные отступы - 16 dp;
расстояние между заголовком и карточками - 12 dp;
расстояние между карточками - 12 dp.
```

Адаптировать под существующую дизайн-систему.

## 7.3. Выбор трёх поездок

Показать до трёх случайно выбранных поездок.

Не выполнять обычный `shuffled()` при каждой recomposition. Иначе карточки будут неожиданно меняться при:

* прокрутке;
* возврате на экран;
* изменении состояния;
* обновлении одной фотографии;
* повороте устройства.

Использовать стабильную случайную выборку.

Рекомендуемый вариант:

```text
seed =
локальная текущая дата
+
версия списка поездок
```

Это означает:

* в течение одного дня пользователь видит одни и те же три поездки;
* на следующий день набор может измениться;
* после добавления или удаления поездки набор может обновиться.

Выбирать только поездки:

* со статусом `CONFIRMED`;
* с корректными датами;
* содержащие хотя бы одну доступную фотографию;
* не помеченные пользователем как скрытые.

Если подтверждение поездок ещё не реализовано, использовать все поездки, кроме удалённых и отклонённых.

Если поездок меньше трёх:

* показать доступное количество;
* не добавлять пустые карточки-заглушки.

## 7.4. Представление карточек

Использовать горизонтальный список:

```text
LazyRow
```

или существующий аналог.

Рекомендуемая ширина карточки:

```text
75-85% ширины экрана
```

На экране должна быть частично видна следующая карточка, чтобы пользователь понимал, что список прокручивается горизонтально.

## 7.5. Содержимое карточки поездки

Карточка должна содержать:

* обложку;
* название поездки;
* диапазон дат;
* количество фотографий;
* количество активных дней;
* количество мест назначения;
* расстояние от домашнего региона, если рассчитано.

Пример:

```text
Казань

10-19 июня 2026
25 фотографий · 7 дней · 3 места
```

Дополнительный badge:

```text
Поездка
```

Для маршрутной поездки:

```text
Маршрут
```

## 7.6. Обложка поездки

Приоритет выбора:

1. Обложка, выбранная пользователем.
2. Фотография с высоким локальным рейтингом качества.
3. Первая доступная фотография поездки.
4. Локальный placeholder.

Не загружать оригинал. Использовать миниатюру.

## 7.7. Действие карточки

При нажатии открывать:

```text
TripDetailsScreen(tripId)
```

Передавать только стабильный ID поездки.

Не передавать:

* список фотографий;
* Bitmap;
* FeatureCollection;
* полную модель поездки через navigation arguments.

## 7.8. Состояния блока поездок

### Loading

Показать три skeleton-карточки.

### Empty

```text
Поездки пока не найдены
Они появятся после анализа фотографий с датой и геопозицией
```

Можно добавить действие:

```text
Проверить фотографии
```

### BackgroundProcessing

Если анализ выполняется в WorkManager:

```text
Ищем поездки в вашей фотогалерее
```

Не блокировать прокрутку страницы.

### Error

```text
Не удалось загрузить поездки
```

Добавить повторную загрузку данных из Room, но не запускать полный анализ автоматически.

# 8. Экран «Все поездки»

## 8.1. Навигация

Переход осуществляется по кнопке:

```text
Все поездки
```

Route:

```text
all_trips
```

Экран должен содержать:

1. Верхнюю панель с кнопкой назад.
2. Карту поездок.
3. Список поездок.
4. Фильтры и сортировку.

## 8.2. Карта поездок

В верхней части экрана «Все поездки» разместить карту высотой:

```text
30% доступной высоты экрана
```

Карта показывает:

* центры одноцентровых поездок;
* места назначения многоцентровых поездок;
* линии между последовательными местами, если они рассчитаны;
* отдельные маркеры поездок.

Не отображать здесь все отдельные фотографии.

## 8.3. Различие между картами

Верхняя карта главной страницы:

```text
показывает тепловую интенсивность всей истории
```

Карта на экране всех поездок:

```text
показывает конкретные поездки и места назначения
```

Не использовать два визуально одинаковых слоя.

## 8.4. Нажатие на маркер

При нажатии:

* выделить соответствующую поездку в списке;
* прокрутить список к карточке;
* либо открыть компактный bottom sheet поездки.

Повторное нажатие или действие «Подробнее» открывает `TripDetailsScreen`.

## 8.5. Список поездок

Под картой показывать поездки вертикальным списком.

Сортировка по умолчанию:

```text
сначала новые
```

Фильтры:

* все;
* подтверждённые;
* возможные;
* одно место;
* маршрутные;
* год;
* длительность.

Не добавлять все фильтры в MVP, если их ещё нет в проекте. Минимально реализовать:

```text
Все
Подтверждённые
Возможные
```

# 9. Карта поездок на главной странице

После карточек «Ваши поездки» добавить отдельную карту.

## 9.1. Размер

```text
30% доступной высоты экрана
```

Минимальная высота:

```text
200 dp
```

## 9.2. Назначение

Карта показывает:

* последние поездки;
* три поездки, показанные в карточках;
* либо все подтверждённые поездки.

Для первой версии рекомендуется показывать все подтверждённые поездки, но визуально выделять три поездки из карточек.

## 9.3. Заголовок

Над картой:

```text
Поездки на карте
```

Дополнительное действие:

```text
Развернуть
```

При нажатии открывается `AllTripsScreen`.

## 9.4. Управление жестами

Поскольку карта находится внутри вертикально прокручиваемой страницы, избежать конфликта жестов.

Рекомендуемое поведение:

* вертикальный свайп одним пальцем прокручивает главную страницу;
* взаимодействие с картой активируется после нажатия;
* либо карта поддерживает масштабирование двумя пальцами;
* после выбора карты пользователь может перемещать её одним пальцем;
* должна быть доступна кнопка выхода из активного режима.

Допустимый упрощённый вариант для MVP:

* отключить перемещение карты на главной;
* оставить только нажатие на маркеры;
* по нажатию на свободную область открывать полноэкранную карту.

Это предпочтительнее, чем неудобный конфликт двух вертикальных скроллов.

# 10. Блок «Ваши места»

## 10.1. Назначение

Блок показывает часто посещаемые или значимые места пользователя.

Примеры:

* домашний город;
* любимый район;
* парк;
* кофейня;
* дача;
* университет;
* частое место прогулок;
* достопримечательность;
* природный объект.

## 10.2. Размер

Минимальная высота блока:

```text
30% доступной высоты экрана
```

Внутри использовать горизонтальный список карточек.

Если содержимое карточек не помещается в 30% высоты, блок может быть выше. Не обрезать текст или фотографии только ради строгого ограничения.

## 10.3. Заголовок

```text
Ваши места                       Все места
```

Кнопка `Все места` открывает:

```text
AllPlacesScreen
```

Если отдельный экран всех мест пока не реализуется, кнопку можно скрыть, но навигационный route и архитектурный задел должны быть предусмотрены.

## 10.4. Какие места показывать

По умолчанию показать до 5 мест.

Сортировка:

```text
placeScore DESC
```

Пример расчёта значимости:

```text
уникальные дни посещения
+
число отдельных визитов
+
число активных месяцев
+
небольшой вес количества фотографий
```

Не сортировать исключительно по количеству фотографий.

Не показывать:

* места без доступных фотографий;
* отклонённые пользователем места;
* удалённые места;
* технические кластеры;
* транзитные точки с низкой уверенностью.

Домашний регион можно показывать, если это соответствует концепции продукта. Добавить возможность скрыть его в настройках приватности.

## 10.5. Карточка места

Карточка должна содержать:

* обложку;
* название;
* количество посещений;
* количество фотографий;
* дату последнего посещения;
* тип места, если определён.

Пример:

```text
Шарташ

18 посещений
246 фотографий
Последний раз - 12 июля
```

Дополнительные badge:

```text
Частое место
Природа
Город
Дом
```

Тип места не должен быть обязательным.

## 10.6. Размер карточки

Рекомендуемая ширина:

```text
55-70% ширины экрана
```

Карточки могут быть меньше карточек поездок.

Обложка должна занимать основную визуальную часть карточки.

## 10.7. Действие

При нажатии открыть:

```text
PlaceDetailsScreen(placeId)
```

Передавать только `placeId`.

## 11.2. Верхняя часть

Показывать:

# 15. Работа с MapLibre

На главной странице одновременно присутствуют две карты:

1. Тепловая карта.
2. Карта поездок.

Не использовать один экземпляр `MapView` одновременно в двух контейнерах.

Каждая карта должна иметь отдельный жизненный цикл.

Обязательно корректно прокидывать:

```text
onStart
onResume
onPause
onStop
onLowMemory
onDestroy
```

или эквивалентную lifecycle-интеграцию Compose.

Не создавать новый MapView при каждой recomposition.

Использовать:

* `remember`;
* `DisposableEffect`;
* существующий lifecycle wrapper;
* существующий MapLibre component проекта.

Источники и слои добавлять один раз после загрузки style.

Обновлять только GeoJSON.

# 25. Ручные сценарии проверки

## Сценарий 1. Есть много поездок и мест

Ожидается:

* heatmap отображается;
* выбраны три поездки;
* видны маркеры поездок;
* отображаются частые места;

Ожидается:

* показывается одна карточка;
* нет пустых карточек;
* карта отображает одну поездку.

Ожидается:

* heatmap показывает empty state;
* блок поездок показывает объяснение;
* блок мест продолжает работать, если места существуют.

## Сценарий 4. Мест нет

Ожидается:

* блок поездок работает;
* блок мест показывает empty state;
* страница не падает.

## Сценарий 5. Анализ выполняется

Ожидается:

* отображается статус фоновой обработки;
* доступные старые результаты остаются видимыми;
* экран не блокируется.

Ожидается:

* вместо миниатюры появляется placeholder;
* карточка не вызывает падение;
* после фонового обновления данные корректируются.

## Сценарий 7. Большая фотобиблиотека

Ожидается:

* прокрутка плавная;
* карты не пересоздаются;
* миниатюры загружаются лениво.

5. Блок «Этот день год назад на карте»

На макете нижний блок имеет большую высоту и занимает почти всю ширину.

Название лучше сделать более универсальным:

Этот день в прошлом

Потому что фотографии могут быть:

год назад;
два года назад;
пять лет назад;
в этот же день в разные годы.

Подзаголовок:

18 июля в разные годы
Размер
высота - 280-420 dp

или:

val onThisDayHeight = screenHeight * 0.42f

Минимальная высота:

300 dp
Содержимое

На карте показываются фотографии, сделанные:

в тот же день и месяц
в предыдущие годы

Например, текущая дата:

18 июля 2026

На карту попадают фотографии:

18 июля 2025
18 июля 2024
18 июля 2023
...

Не включать фотографии текущего года, если нужен именно режим воспоминаний.

Маркеры

На карте показывать:

одиночные миниатюры;
кластеры фотографий;
год рядом с маркером;
количество фотографий в точке.

Пример:

2023 · 12 фото

7. Итоговая структура Compose
@Composable
fun HomeScreen(
    state: HomeUiState,
    onOpenMap: () -> Unit,
    onOpenAllTrips: () -> Unit,
    onOpenTrip: (Long) -> Unit,
    onOpenAllPlaces: () -> Unit,
    onOpenPlace: (Long) -> Unit,
    onOpenOnThisDay: (Int, Int) -> Unit
) {
    LazyColumn {
        item {
            HomeHeatmapSection(...)
        }

        item {
            FeaturedTripsSection(...)
        }

        item {
            PopularPlacesSection(...)
        }

        item {
            OnThisDayMapSection(...)
        }
    }
}

Карты нельзя создавать заново при каждой recomposition. Для каждого MapLibre-компонента использовать стабильный экземпляр и корректную lifecycle-интеграцию.

8. Критерии приёмки по макету
сверху расположена тепловая карта;
карта занимает около трети высоты экрана, а не 60vh;
ниже находится горизонтальный список поездок;
одновременно видны две карточки и часть третьей;
есть переход Все поездки;
ниже расположен полноширинный блок популярных мест;
карточки мест открывают страницу места;
страница места показывает мини-галереи по дням и месяцам;
последним расположен большой блок Этот день в прошлом;
блок воспоминаний содержит отдельную карту;
карта показывает фотографии за тот же день прошлых лет;
главная страница прокручивается вертикально;
встроенные карты не мешают вертикальной прокрутке;
все вычисления поездок, мест и воспоминаний выполняются до отображения страницы;
главная читает готовые данные из Room.
