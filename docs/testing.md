# План тестирования Traverse

Этот документ описывает, как тестировать текущий Android-проект **Traverse**.
Он не является заданием на массовый рефакторинг. Его задача - зафиксировать
приоритеты тестирования, границы автоматических проверок и ручные сценарии.

Важно для Codex: после обычного пользовательского промпта не запускать тесты,
Gradle-проверки, `npx`, `tsc`, `eslint` и похожие команды. Если проверку нужно
выполнить, сначала явно согласовать это с пользователем. В ответе обязательно
указать, какие проверки не запускались.

---

## 1. Текущий контекст проекта

Traverse - нативное Android-приложение на Kotlin и Jetpack Compose.

Сейчас в проекте есть:

* один Android-модуль `:app`;
* пакет приложения `com.example.photomap`;
* Jetpack Compose, Material 3 и Navigation Compose;
* runtime-разрешения для фото на Android 10+;
* чтение фотографий через `MediaStore.Images.Media`;
* чтение GPS из MediaStore и EXIF через `ExifInterface`;
* локальный индекс `photo_index.db` на `SQLiteOpenHelper`;
* кластеризация фотографий для основной карты;
* MapLibre Android SDK;
* Compose-overlay маркеры поверх карты;
* дата-фильтр карты;
* локальная сегментация поездок;
* карта поездок и экран деталей поездки;
* trip heatmap на H3-ячейках с fallback-сеткой;
* главная страница с heatmap preview, поездками, местами и воспоминаниями;
* начальная семантика мест в `HomePlaceSemantics`;
* `TripHeatmapWorker` для фонового пересчёта поездок/теплокарты по уже
  проиндексированным фото.

Сейчас в проекте нет:

* Room;
* DAO в смысле Room;
* Hilt или другого DI-фреймворка;
* Coil;
* полноценного фонового сканирования MediaStore через WorkManager;
* персистентной таблицы places;
* сетевого геокодинга мест как основной функции;
* отдельной тестовой инфраструктуры для coroutines, Flow, WorkManager или Room.

Поэтому тест-план должен проверять текущий код, а не требовать сразу
несуществующую архитектуру.

---

## 2. Главные цели тестирования

1. Защитить текущее пользовательское поведение перед следующими изменениями.
2. Зафиксировать инварианты карты, кластеризации, поездок, trip heatmap и
   локального индекса.
3. Отделять чистую бизнес-логику от Android API там, где это нужно для тестов.
4. Не проверять внутренности MapLibre, MediaStore, Android framework и внешних
   tile provider.
5. Не трогать реальные фотографии пользователя в автоматических тестах.
6. Не считать 100% coverage целью само по себе.
7. Явно отделять автоматические тесты от ручной проверки на устройстве.

---

## 3. Ограничения

В тестах нельзя:

* читать реальную галерею пользователя;
* обращаться в интернет;
* зависеть от OpenFreeMap, MapLibre tile loading или внешнего geocoder;
* использовать production-базу `photo_index.db`;
* проверять private-методы напрямую;
* копировать реализацию тестируемого алгоритма в assertions;
* использовать `Thread.sleep`;
* зависеть от системного времени без контролируемых часов;
* создавать случайные данные без фиксированного seed;
* проверять внутреннюю реализацию MapLibre SDK;
* передавать реальные `Bitmap`, MapLibre objects или большие списки через
  навигацию ради теста.

В рамках текущего проекта не планировать Room-тесты как обязательные. Если
принимается отдельное решение мигрировать индекс на Room, тогда добавляется
отдельный план миграции и тесты Room migration.

---

## 4. Пирамида тестов

### Unit-тесты

Unit-тесты должны покрывать код без Android framework или с минимальными fake:

* `LocationValidator`;
* `GeoCoordinates`;
* `PhotoDateFilter` и `matchesPhotoDateFilter`;
* builders и нормализацию настроек кластеризации;
* `PhotoClusterBuilder`;
* `PhotoMapFeatureMapper`;
* `PhotoHeatmapFeatureMapper`;
* `TripHeatmapFeatureMapper`;
* `PhotoMapRenderRules`;
* `PhotoMapClickDecision`;
* `PhotoMapDebugInfo`;
* `HeuristicTripSegmenter`;
* `TripHeatmapBuilder`;
* `tripHeatIntensity`;
* `HomePlaceSemantics`;
* reducer-like части `PhotoAccessViewModel`, если они будут вынесены из
  AndroidViewModel.

### Integration-тесты

Integration-тесты проверяют связку нескольких компонентов:

* `PhotoIndexDatabase` на временной SQLite-базе;
* schema creation и upgrade для `photo_index.db`;
* сохранение и чтение фотографий;
* сохранение и чтение кластеров;
* сохранение и чтение поездок;
* сохранение и чтение trip heatmap;
* удаление отсутствующих MediaStore ID;
* `PhotoClusterStore`;
* `TripStore` с fake `TripSegmenter`;
* `TripHeatmapStore`;
* `AndroidMediaStorePhotoReader` только после появления seam для fake
  `ContentResolver`, fake базы и fake EXIF reader.

### Instrumentation/UI-тесты

UI-тесты должны проверять пользовательские сценарии через Compose semantics:

* экран разрешений;
* отказ в доступе к фото;
* частичный доступ к фото;
* открытие главной после доступных разрешений;
* переход на основную карту;
* отображение пустого состояния карты;
* открытие настроек;
* изменение настроек кластеров;
* переход на карту поездок;
* открытие деталей поездки;
* работа `MiniPhotoGallery`;
* переход из места на список фото;
* возврат с экрана деталей с сохранением ожидаемого состояния.

MapLibre в UI-тестах проверять только по внешнему поведению: экран не падает,
контейнер карты отображается, overlay/semantics обновляются, пользовательские
переходы работают. Содержимое тайлов и внутренние style layers не являются
стабильной зоной assertions.

---

## 5. Тестовая инфраструктура

Текущие зависимости покрывают только базовый JUnit и Android/Compose
instrumentation. Перед расширением тестов отдельно рассмотреть добавление:

* `kotlinx-coroutines-test` для детерминированных dispatcher;
* `app.cash.turbine` или простой локальный collector для `Flow`;
* `androidx.work:work-testing` для Worker;
* Robolectric, если нужно тестировать Android-зависимый код без устройства;
* Room testing только если проект действительно переходит на Room;
* fixtures/builders для `DevicePhoto`, `IndexedPhoto`, `PhotoMapBounds`,
  `DetectedTrip`, `DetectedTripDestination`, `TripSegmentationResult`.

Минимальные helpers:

```kotlin
fun testDevicePhoto(
    mediaId: Long = 1L,
    uri: String = "content://media/external/images/media/1",
    dateTaken: Long? = 1_700_000_000_000L,
    latitude: Double? = 56.8389,
    longitude: Double? = 60.6057
): DevicePhoto
```

```kotlin
fun testBounds(
    south: Double = 56.0,
    north: Double = 57.0,
    west: Double = 60.0,
    east: Double = 61.0
): PhotoMapBounds
```

```kotlin
interface AppClock {
    fun currentTimeMillis(): Long
    fun elapsedRealtimeMillis(): Long
}
```

`AppClock` нужен, потому что сейчас часть кода использует
`System.currentTimeMillis()` и `SystemClock.elapsedRealtime()`.

## 6. Обязательные unit-тесты

### Координаты и даты

Проверить:

1. Невалидные широта/долгота отбрасываются.
2. Координата `(0, 0)` не считается пользовательской геоточкой, если это
   правило сохранено в `LocationValidator`.
3. `photoDateMillis()` выбирает дату в ожидаемом порядке.
4. Дата-фильтр включает границы диапазона.
5. Дата-фильтр корректно работает на пустом наборе.
6. Группировка по дням и месяцам стабильна при одинаковых timestamp.

### Кластеризация основной карты

Проверить:

1. Фото без координат не попадает в кластеры.
2. Одна фотография с координатами остаётся видимой как одиночный кластер.
3. На каждом сохранённом уровне сумма ссылок `PhotoClusterLink` равна числу
   фотографий с координатами.
4. `minPoints` не удаляет одиночные фото, а только влияет на объединение.
5. Плотный городской набор объединяется.
6. Далёкие точки остаются разными кластерами.
7. `parentClusterId` стабилен между уровнями.
8. `coverPhotoId` выбирается по самой свежей доступной дате.
9. Изменение `PhotoClusterSettings` меняет результат предсказуемо.
10. Алгоритм проверяет отмену через coroutine cancellation.

### Основная карта и overlay

Проверить:

1. `PhotoMapFeatureMapper` создаёт корректные GeoJSON features.
2. `PhotoHeatmapFeatureMapper` не создаёт features для пустого набора.
3. `TripHeatmapFeatureMapper` нормализует веса в ожидаемом диапазоне.
4. `PhotoMapRenderRules` выбирает правильный режим маркеров по zoom и настройкам.
5. `PhotoMapClickDecision` игнорирует случайный tap после жеста карты.
6. `PhotoMapDebugInfo` не раскрывает приватные данные вроде полного URI или имени
   файла, если это не debug-only поведение.

### Сегментация поездок

Проверить:

1. Фото без даты или координат попадают в `skippedPhotos`.
2. Домашние фотографии не создают поездку.
3. Один удалённый away-сегмент создаёт поездку.
4. Несколько away-регионов подряд создают `MULTI_DESTINATION_TRIP`.
5. Возврат к базовому региону завершает поездку.
6. Пробел 1-2 дня внутри away-сегмента не разрывает поездку.
7. Длинный неизвестный пробел с другим регионом разрывает сегмент.
8. Слишком короткая или слишком близкая к базе поездка отбрасывается.
9. Низкая confidence не создаёт подтверждённую поездку.
10. Повторный расчёт на тех же данных даёт стабильные trip ID.
11. Удаление фотографии меняет границы или состав поездки предсказуемо.
12. Большой набор фотографий не блокирует cancellation.

### Trip heatmap

Проверить:

1. Пустой список поездок даёт пустую теплокарту.
2. Домашние/base destinations исключаются из trip heatmap.
3. Away destination создаёт вклад на всех ожидаемых H3-resolution.
4. Fallback-сетка работает, если H3 недоступен.
5. `tripHeatIntensity` монотонно растёт при росте `tripCount`,
   `daysSpent`, `activeDays`, `sessionCount`.
6. `dataVersion` управляется входным параметром, а не случайным временем в тесте.
7. Вклад одной поездки не дублируется в одной и той же ячейке/resolution.
8. Фото без даты или координат не ломают построение heatmap.
9. Несколько destination одной поездки дают разные contributions, если они в
   разных ячейках.
10. Алгоритм проверяет cancellation на больших входных данных.

### Места на главной

Текущие места - это не полноценный каталог и не Room-таблицы. Сейчас нужно
тестировать `HomePlaceSemantics`.

Проверить:

1. `place=city` принимается как `City`.
2. Административный район принимается как `District`.
3. Именованный парк принимается как `NaturalArea`.
4. `landuse=residential` отклоняется как technical.
5. `building`, `highway`, `power`, `utility`, `route` отклоняются.
6. Техническое название вроде `городская территория 200B` отклоняется.
7. Объект без имени отклоняется.
8. Объект с некачественным именем получает причину отказа.
9. Один `source + sourceObjectId` создаёт один стабильный `geoObjectId`.
10. Фотография может иметь несколько связей с геообъектами.
11. Родственные места определяются через граф связей, а не через строковое имя.
12. Unsupported, unknown и technical статусы не смешиваются.

### Настройки

Проверить:

1. `PhotoClusterSettings.normalized()` ограничивает значения min/max.
2. Шаги изменения настроек не выводят значения за допустимые границы.
3. Значения по умолчанию соответствуют `core.settings`.
4. Настройки, влияющие на вид карты, не меняют исходный список фотографий.

## 7. Обязательные integration-тесты

### SQLite-индекс

Использовать временную тестовую базу, а не production `photo_index.db`.

Проверить:

1. `onCreate` создаёт таблицы `photos`, `photo_clusters`,
   `photo_cluster_links`, `photo_cluster_meta`, `trips`, `trip_photo_links`,
   `trip_destinations`, `trip_heat_cells`, `trip_heat_contributions`,
   `trip_heat_meta`.
2. `onUpgrade` корректно проходит пути `1 -> 5`, `2 -> 5`, `3 -> 5`, `4 -> 5`.
3. `upsertPhotos` идемпотентен.
4. Изменение `dateModified` или `size` обновляет запись.
5. `deleteMissingPhotos` удаляет отсутствующие фото и связанные ссылки
   кластеров/поездок.
6. `getPhotosInBounds` фильтрует по viewport.
7. `getPhotosInBounds` применяет `PhotoDateFilter`.
8. `replaceClusters` заменяет старое поколение кластеров атомарно.
9. `replaceTrips` заменяет поездки, destinations и links атомарно.
10. `replaceTripHeatmap` заменяет heat cells и contributions атомарно.
11. Индексы создаются для частых запросов viewport/date.
12. Ошибка в середине транзакции не оставляет частично записанное состояние.

### Stores

Проверить:

1. `PhotoClusterStore.rebuildClusters` сохраняет кластеры и links.
2. `PhotoClusterStore.loadVisibleMapContent` возвращает только видимый viewport.
3. `PhotoClusterStore` использует расширенные bounds, если это предусмотрено
   текущей реализацией.
4. `TripStore.rebuildTrips` с fake `TripSegmenter` сохраняет trips и heatmap.
5. `TripStore.loadTripMarkers` возвращает сохранённые маркеры.
6. `TripStore.getTripPhotoIdsByTripIds` не теряет пустые поездки.
7. `TripHeatmapStore.loadVisibleHeatmap` выбирает resolution по zoom.
8. `TripHeatmapStore.loadVisibleHeatmap` нормализует веса локально для viewport.

### MediaStore/EXIF

Текущий `AndroidMediaStorePhotoReader` сильно завязан на `ContentResolver`,
`PhotoIndexDatabase`, `PhotoPermissionManager`, `SystemClock` и EXIF reader.
Перед подробными unit/integration-тестами желательно ввести seams:

* интерфейс для MediaStore query;
* интерфейс для EXIF location reader;
* injectable `PhotoIndexDatabase` или repository;
* injectable permission checker;
* `AppClock`.

После этого проверить:

1. Новая фотография создаёт одну запись.
2. Повторное сканирование не создаёт дубликат.
3. Неизменённая фотография переиспользует сохранённую координату.
4. Изменение ревизии заставляет перечитать location.
5. EXIF scan resume state сохраняется при паузе/отмене.
6. Resume state очищается после успешного полного EXIF scan.
7. SecurityException не роняет приложение.
8. RuntimeException одной операции не теряет уже сохранённый batch.
9. `PersistBatchSize` не влияет на итоговый набор.
10. Удалённые из MediaStore фото удаляются из локального индекса.

### WorkManager

Текущий `TripHeatmapWorker` не является полноценным worker-сканером MediaStore.
Он пересчитывает trips/heatmap по индексированным фото.

Проверить:

1. Нет доступа к фото -> worker завершается `success` без падения.
2. Корректный пересчёт -> `success`.
3. Временная RuntimeException -> `retry`.
4. `CancellationException` пробрасывается.
5. `enqueueOneTime` использует unique work с `KEEP`.
6. `enqueuePeriodic` использует unique periodic work с `UPDATE`.

Для unit-тестируемости worker желательно сначала вынести фабрики
`DevicePhotoReader` и `TripStore`.

## 8. Обязательные UI-сценарии

Автоматизировать только то, что стабильно через Compose semantics. Остальное
оставить ручной проверкой.

### Разрешения и старт

Проверить:

1. Первый запуск без разрешений показывает экран запроса доступа.
2. Отказ в доступе показывает понятное состояние и не запускает сканирование.
3. Полный доступ открывает главную страницу.
4. Частичный доступ на Android 14+ не ломает навигацию.
5. Повторное возвращение из настроек обновляет permission status.

### Главная

Проверить:

1. Экран прокручивается вертикально.
2. Heatmap preview не запускает тяжёлый пересчёт trips на UI thread.
3. Карточка поездки открывает экран деталей.
4. `Все поездки` открывает карту поездок.
5. Карточка места открывает детали места.
6. Пустые данные показывают empty state без падения.
7. `MiniPhotoGallery` не ломается на 0, 1 и большом числе фото.

### Основная карта

Проверить:

1. Переход на карту доступен с главной.
2. Empty state карты отображается без падения.
3. Дата-фильтр меняет видимые overlay-маркеры.
4. Смена настроек кластеров обновляет overlay.
5. Tap после жеста карты не открывает случайную галерею.
6. Debug panel включается и выключается из настроек.

### Карта поездок и детали поездки

Проверить:

1. Карта поездок открывается.
2. Tap по поездке открывает детали поездки.
3. Детали поездки показывают дату, название и grid фото.
4. Tap по фото открывает системный viewer через intent, если сценарий
   автоматизируем стабильно.
5. Возврат с деталей сохраняет ожидаемую позицию карты или выбранную поездку.
6. Scrubber поездок/фото меняет selection без пересчёта heatmap.

---

## 9. Ручная проверка на устройстве

Эти сценарии трудно стабильно покрыть автоматикой, поэтому их нужно держать как
ручной smoke checklist:

1. Первый запуск на Android 10-12.
2. Первый запуск на Android 13.
3. Первый запуск на Android 14+ с partial photo access.
4. Галерея 0 фото.
5. Галерея без GPS.
6. Галерея 1 фото с GPS.
7. Галерея 500+ фото.
8. Галерея 10 000+ фото, если доступен тестовый набор.
9. EXIF scan с паузой, продолжением и отменой.
10. Возврат в приложение после изменения разрешений в системных настройках.
11. Карта без интернета или с недоступным style URL.
12. Быстрые жесты карты: drag, pinch, double tap.
13. Быстрые taps по маркерам сразу после жеста.
14. Смена ориентации экрана, если приложение её поддерживает.
15. Устройство с низкой производительностью: отсутствие ANR и заметных фризов.
16. Проверка, что приложение не изменяет и не удаляет пользовательские фото.

## 10. Рекомендуемый порядок работ

### Этап 1. Аудит и фиксация текущих контрактов

* описать текущую структуру пакетов;
* перечислить Android-зависимые участки;
* перечислить уже существующие тесты;
* определить классы, где для тестов нужны seams;
* зафиксировать список файлов, которые будут изменяться.

Код не менять до конца аудита.

### Этап 2. Тестовые данные и чистые unit-тесты

* добавить builders/fixtures;
* покрыть `core.util`;
* покрыть `PhotoDateFilter`;
* расширить тесты map helper-классов;
* покрыть `HomePlaceSemantics`.

### Этап 3. Алгоритмы карты, кластеров, поездок и heatmap

* покрыть `PhotoClusterBuilder`;
* покрыть `HeuristicTripSegmenter`;
* покрыть `TripHeatmapBuilder`;
* покрыть feature mappers;

## 11. Критерии готовности тестового покрытия

Минимально достаточное состояние:

* чистые алгоритмы имеют unit-тесты на основные инварианты;
* SQLite-контракт покрыт integration-тестами;
* основные маршруты Compose покрыты UI smoke-тестами;
* MapLibre не тестируется как SDK, но обёртки/мапперы покрыты;
* MediaStore и EXIF не читают реальные файлы в автоматических тестах;
* состояние разрешений проверяется через fake или instrumentation;
* тесты не зависят от сети, системного времени и порядка запуска;
* любые добавленные test dependencies описаны в changelog/PR;
* непокрытые вручную сценарии явно перечислены в отчёте.

Ориентиры по покрытию использовать как вспомогательный сигнал:

```text
core/domain/data algorithms: 80%+
map/trip/heatmap helper-код: 75%+
ViewModel после внедрения seams: 70%+
общее покрытие проекта: не ниже 55-60%
```

Не добавлять бессмысленные тесты ради процента.

## 12. Команды для ручного запуска

Эти команды приведены для человека или для отдельного согласованного шага.
Codex не должен запускать их автоматически после обычного промпта.

```powershell
.\gradlew.bat test
.\gradlew.bat connectedDebugAndroidTest
.\gradlew.bat assembleDebug
```

При необходимости Android lint:

```powershell
.\gradlew.bat lintDebug
```

Если проверка не запускалась, итоговый отчёт должен прямо сказать:

```text
Автоматические проверки не запускались по инструкции проекта.
```
