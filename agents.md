# Техническое задание: Android-приложение «Фотографии на карте»

## 1. Общая задача

Необходимо разработать нативное Android-приложение-галерею, которое получает фотографии из памяти устройства и отображает снимки с географическими координатами на интерактивной карте.

По принципу работы приложение должно быть похоже на раздел карты в Google Photos:

* фотографии отображаются в виде меток с миниатюрами;
* расположенные рядом фотографии объединяются в кластеры;
* при приближении карты кластеры разделяются на отдельные фотографии;
* при нажатии на метку открывается фотография или список фотографий в этой точке;
* приложение работает локально, без регистрации и собственного сервера.

## 2. Основной стек

## 4. Основные экраны

### 4.1. Экран запроса разрешений

При первом запуске приложение должно объяснять, зачем требуется доступ к фотографиям.

На экране должны быть:

* заголовок;
* краткое описание;
* кнопка «Предоставить доступ»;
* обработка отказа;
* возможность повторно запросить разрешение;
* кнопка перехода в системные настройки, если пользователь запретил повторный запрос.

Необходимо учитывать различия разрешений между версиями Android.

Для чтения фотографий использовать соответствующее системное разрешение в зависимости от версии Android.

Для чтения исходных географических координат использовать `ACCESS_MEDIA_LOCATION`, если оно требуется системой.

### 4.2. Экран карты

Главный экран приложения.

На экране должны отображаться:

* Google Map;
* метки фотографий;
* кластеры фотографий;
* индикатор загрузки во время первичного сканирования;
* количество найденных фотографий с координатами;
* кнопка перехода к обычной галерее;
* кнопка центрирования карты;
* кнопка повторного сканирования фотографий;
* кнопка перехода к настройкам.

Поведение карты:

* при первом открытии карта должна показывать область, содержащую все найденные фотографии;
* если фотографий с координатами нет, показывать информационное состояние;
* фотографии, расположенные близко друг к другу, должны объединяться в кластеры;
* на кластере должно отображаться количество фотографий;
* при нажатии на кластер карта должна приближаться;
* при нажатии на одиночную метку должна открываться нижняя панель с превью фотографии;
* при нажатии на превью должен открываться полноэкранный просмотр;
* при перемещении карты не нужно повторно читать EXIF;
* метки должны загружаться из локальной базы данных.

Желательно отображать миниатюру фотографии внутри кастомной метки.

Если это сильно усложняет первую реализацию, сначала использовать стандартный маркер, но архитектура должна позволять позже заменить его на маркер с миниатюрой.

### 4.3. Экран галереи

Экран со всеми доступными фотографиями.

Использовать сетку `LazyVerticalGrid`.

Функции:

* отображение миниатюр;
* сортировка по дате съёмки от новых к старым;
* плавная прокрутка;
* ленивое декодирование изображений;
* отображение индикатора геолокации у фотографий с координатами;
* нажатие на фотографию открывает полноэкранный просмотр;
* фотографии без координат также должны отображаться в галерее.

Для загрузки изображений использовать Coil.

Не загружать оригиналы изображений для отображения сетки. Использовать уменьшенные изображения и кеширование.

### 4.4. Полноэкранный просмотр фотографии

Экран должен содержать:

* фотографию;
* кнопку возврата;
* дату съёмки;
* координаты, если они существуют;
* имя файла;
* кнопку «Показать на карте», если у фотографии есть координаты.

Дополнительно:

* поддержать масштабирование жестом;
* поддержать перелистывание соседних фотографий;
* не изменять и не удалять оригинальный файл.

### 4.5. Экран настроек

Добавить следующие настройки:

* запуск повторного сканирования;

* отображение количества проиндексированных фотографий;
* отображение количества фотографий с координатами;

* информация о приложении;
* версия приложения.

## 5. Работа с фотографиями

Получать фотографии через Android `MediaStore`.

Для каждой фотографии получить:

* MediaStore ID;
* URI;
* имя файла;
* MIME-тип;
* дату добавления;
* дату изменения;
* дату съёмки;
* ширину;
* высоту;
* размер файла;
* ориентацию;
* широту;
* долготу;
* наличие географических координат.

Не использовать абсолютные файловые пути как основной идентификатор.

Основной идентификатор фотографии - MediaStore ID.

URI должен формироваться через `ContentUris.withAppendedId`.

## 6. Чтение координат

Для определения координат использовать следующий порядок:

1. Попытаться получить координаты из доступных колонок MediaStore, если они доступны.
2. Если координаты отсутствуют, прочитать EXIF через `ExifInterface`.
3. Для открытия изображения использовать `ContentResolver`.
4. Корректно обрабатывать отсутствие доступа, повреждённые файлы и неподдерживаемые форматы.

Фотографии без координат не должны отображаться на карте, но должны оставаться в общей галерее.

Координаты считать существующими только при корректных значениях:

* широта от -90 до 90;
* долгота от -180 до 180;
* значения не равны одновременно 0 и 0.

## 7. Локальная база данных

Использовать Room.

Создать сущность `PhotoEntity`.

Пример структуры:

```kotlin
@Entity(
    tableName = "photos",
    indices = [
        Index(value = ["dateTaken"]),
        Index(value = ["latitude"]),
        Index(value = ["longitude"]),
        Index(value = ["hasLocation"])
    ]
)
data class PhotoEntity(
    @PrimaryKey
    val mediaId: Long,
    val uri: String,
    val displayName: String?,
    val mimeType: String?,
    val dateAdded: Long?,
    val dateModified: Long?,
    val dateTaken: Long?,
    val width: Int?,
    val height: Int?,
    val size: Long?,
    val orientation: Int?,
    val latitude: Double?,
    val longitude: Double?,
    val hasLocation: Boolean
)
```

Создать DAO с методами:

```kotlin
fun observeAllPhotos(): Flow<List<PhotoEntity>>

fun observePhotosWithLocation(): Flow<List<PhotoEntity>>

suspend fun getPhotoById(mediaId: Long): PhotoEntity?

suspend fun getPhotosInBounds(
    minLatitude: Double,
    maxLatitude: Double,
    minLongitude: Double,
    maxLongitude: Double
): List<PhotoEntity>

suspend fun deleteMissingPhotos(existingMediaIds: List<Long>)

suspend fun clearAll()

suspend fun getPhotoCount(): Int

suspend fun getPhotosWithLocationCount(): Int
```

Для большого количества ID не формировать один слишком большой SQL-запрос. Удаление отсутствующих фотографий реализовать безопасным способом, например через временную таблицу, пакетную обработку или сравнение данных на уровне репозитория.

## 8. Сканирование медиатеки

Создать отдельный компонент:

```kotlin
interface PhotoScanner {
    suspend fun scanPhotos(
        onProgress: (ScanProgress) -> Unit
    ): ScanResult
}
```

Сканирование должно:

* получать изображения через MediaStore;
* обрабатывать фотографии пакетами;
* читать EXIF только при необходимости;
* не блокировать главный поток;
* отправлять прогресс;
* сохранять результаты в Room;
* обновлять существующие записи;
* удалять из базы записи о фотографиях, которых больше нет;
* продолжать работу при ошибке чтения отдельного файла;
* возвращать количество найденных, добавленных, обновлённых и пропущенных фотографий.

Пример состояния:

```kotlin
data class ScanProgress(
    val processed: Int,
    val total: Int,
    val currentFileName: String?
)
```

Пример результата:

```kotlin
data class ScanResult(
    val totalFound: Int,
    val withLocation: Int,
    val withoutLocation: Int,
    val failed: Int
)
```

Во время первого запуска показывать прогресс:

```text
Сканирование фотографий
Обработано 620 из 2400
```

## 9. Фоновое обновление

Использовать WorkManager.

Создать `PhotoScanWorker`.

Требования:

* периодически проверять изменения в медиатеке;
* выполнять работу только при наличии необходимых разрешений;
* не запускать несколько сканирований одновременно;
* не блокировать интерфейс;
* сохранять результат в Room;
* поддержать ручной запуск через настройки;

Для первой версии допустимо периодическое сканирование один раз в сутки.

Не задавать точное время запуска.

## 13. Кластеризация

Для карты использовать кластеризацию из Maps Compose Utils.

Каждая фотография с координатами должна быть представлена объектом, реализующим необходимый интерфейс кластерного элемента.

Пример:

```kotlin
data class PhotoClusterItem(
    val photoId: Long,
    val position: LatLng,
    val photoUri: String
) : ClusterItem {
    override fun getPosition(): LatLng = position
    override fun getTitle(): String? = null
    override fun getSnippet(): String? = null
    override fun getZIndex(): Float? = null
}
```

Требования:

* кластеры должны корректно обновляться после изменения данных;
* на кластере показывать количество фотографий;
* при нажатии на кластер изменять позицию камеры;
* при нажатии на элемент открывать превью;
* не создавать Bitmap полной фотографии для каждой метки;
* кешировать миниатюры маркеров;
* ограничивать размер превью метки.

## 16. Обработка ошибок

Приложение не должно завершаться аварийно при:

* повреждённом изображении;
* отсутствии EXIF;
* неправильных GPS-данных;
* удалении фотографии во время сканирования;
* отсутствии доступа к отдельному URI;
* ошибке декодирования изображения;
* временной недоступности Google Maps;
* отсутствии API-ключа;
* пустой медиатеке;
* отсутствии фотографий с координатами;
* нехватке памяти.

Ошибки отдельных файлов логировать и пропускать.

Для логирования использовать Android Log или отдельную простую обёртку.

Не выводить пользователю технический stack trace.

## 17. Производительность

Приложение должно быть рассчитано минимум на 20 000 фотографий.

Требования:

* не хранить Bitmap в Room;
* не копировать оригинальные изображения;
* не читать EXIF при каждой отрисовке;
* выполнять сканирование на `Dispatchers.IO`;
* сохранять фотографии пакетами;
* использовать Coil memory cache и disk cache;
* использовать LazyGrid;
* не передавать в Compose огромный изменяемый список;
* использовать стабильные ключи;
* не загружать оригиналы для миниатюр;
* избегать создания новых объектов карты при каждой рекомпозиции;
* использовать `remember`;
* использовать `derivedStateOf` там, где это уместно;
* не читать все фотографии из базы при каждом движении карты.

Для карты желательно получать только фотографии, попадающие в текущие границы карты.

Обновление запроса после движения карты выполнять с debounce около 300-500 мс.

## 25. Правила именования веток, коммитов и версионирования

### Ветки

Основная ветка разработки - `main`.

Рабочие ветки создавать от актуальной `main` по формату:

```text
<type>-<number>-<short-slug>
```

Правила:

* использовать только латиницу в нижнем регистре, цифры и дефисы;
* `<type>` выбирать из списка: `init`, `feature`, `fix`, `refactor`, `docs`, `test`, `chore`, `release`, `hotfix`;
* `<number>` задавать двумя цифрами по порядку этапа или задачи: `01`, `02`, `03`;
* `<short-slug>` должен кратко описывать результат ветки, например `project-bootstrap`, `permissions-mediastore`, `room-scanner`;
* не использовать пробелы, подчёркивания, кириллицу и длинные фразы;
* одна ветка должна закрывать один логический этап или одну связанную задачу;
* в каждой рабочей ветке должен быть файл `branch.md` с заданием, границами работ, этапами и критериями готовности этой ветки;
* раздел текущей ветки в `agents.md` должен содержать ссылку на соответствующий `branch.md`.

Примеры:

```text
init-01-project-bootstrap
feature-02-permissions-mediastore
feature-03-room-scanner
feature-04-gallery
feature-05-map-clustering
feature-06-background-scan
docs-07-readme-tests
fix-08-media-location-permission
release-0-1-0
```

### Коммиты

Коммиты оформлять в стиле Conventional Commits:

```text
<type>(<scope>): <краткое действие>
```

Правила:

* `<type>` выбирать из списка: `feat`, `fix`, `docs`, `refactor`, `test`, `build`, `ci`, `chore`, `perf`;
* `<scope>` указывать коротко и по-английски: `app`, `gradle`, `media`, `room`, `map`, `gallery`, `viewer`, `settings`, `docs`;
* описание после двоеточия писать кратко, в повелительной или результативной форме;
* один коммит должен описывать одно логическое изменение;
* не коммитить реальные API-ключи, локальные настройки IDE, временные файлы и сгенерированные артефакты сборки;
* если изменение влияет на поведение пользователя, в теле коммита кратко описывать причину и последствия;
* после каждой завершённой пользовательской команды, если в рабочем дереве есть изменения, делать отдельный коммит по этим правилам;
* не включать в такой коммит чужие, незавершённые или явно не относящиеся к выполненной команде изменения.

Примеры:

```text
build(gradle): add Android app module
feat(app): add Compose navigation shell
feat(media): scan images through MediaStore
fix(room): skip invalid zero coordinates
docs(readme): describe Maps API key setup
```

### Версионирование

Использовать SemVer:

```text
MAJOR.MINOR.PATCH
```

Правила:

* до первого полноценного MVP использовать версии `0.x.y`;
* первая рабочая версия приложения после базовой инициализации - `0.1.0`;
* `MAJOR` увеличивать при несовместимых изменениях архитектуры или пользовательских данных;
* `MINOR` увеличивать при добавлении новых пользовательских возможностей;
* `PATCH` увеличивать при исправлениях ошибок без изменения набора функций;
* Android `versionName` должен совпадать с SemVer;
* Android `versionCode` увеличивать на 1 при каждой сборке, предназначенной для установки вне локальной разработки;
* релизные теги создавать в формате `v<version>`, например `v0.1.0`;
* релизные ветки создавать в формате `release-<version>`, например `release-0-1-0`;
* подробные правила подготовки релизов описаны в [release.md](release.md).

## 28. Критерии готовности MVP

MVP считается готовым, если:

* проект открывается в Android Studio;
* проект успешно собирается;
* приложение запускается на устройстве или эмуляторе;
* пользователь может предоставить доступ к фотографиям;
* фотографии считываются через MediaStore;
* данные сохраняются в Room;
* фотографии отображаются в галерее;
* фотографии с GPS отображаются на карте;
* близкие фотографии объединяются в кластеры;
* фотографию можно открыть в полноэкранном режиме;
* повторный запуск не приводит к полному чтению EXIF без необходимости;
* удалённые фотографии исчезают из локального индекса после сканирования;
* приложение не падает при отсутствии координат;
* приложение не падает при пустой галерее;
* API-ключ Google Maps не хранится в Git;
* основные функции покрыты тестами;
* README содержит инструкцию по запуску.
