Для приложения только под Android я бы выбрал нативный стек Kotlin + Jetpack Compose. Для работы с системной галереей, EXIF-координатами, разрешениями и большим количеством фотографий он будет удобнее и надёжнее React Native или Flutter.

Рекомендуемый стек
Задача	Технология
Язык	Kotlin
Интерфейс	Jetpack Compose + Material 3
Карта	Google Maps Compose
Метки и кластеры	Maps Compose Utils / Clustering
Получение фотографий	Android MediaStore
Получение GPS из фото	ExifInterface
Локальная база	Room
Загрузка миниатюр	Coil 3
Асинхронность	Kotlin Coroutines + Flow
Состояние экранов	ViewModel + StateFlow
Фоновое сканирование	WorkManager
Dependency Injection	Hilt, опционально
Навигация	Navigation Compose

Jetpack Compose является рекомендуемым современным инструментом Google для разработки нативных Android-интерфейсов. Официальная библиотека Maps Compose позволяет использовать карту, маркеры и остальные элементы непосредственно внутри Compose.

Почему лучше делать нативно

Приложению придётся тесно работать с Android:

запрашивать доступ к фотографиям;
читать системную медиатеку через MediaStore;
получать настоящие GPS-координаты из EXIF;
реагировать на добавление и удаление фотографий;
генерировать и кешировать миниатюры;
индексировать тысячи файлов в фоне;
учитывать разные правила доступа к фото в разных версиях Android.

Для чтения неотредактированных координат из EXIF на Android 10 и новее требуется разрешение ACCESS_MEDIA_LOCATION. Сами фотографии удобно получать через MediaStore.

Во Flutter и React Native значительную часть этой логики всё равно пришлось бы реализовывать через Android-плагины или писать собственные Kotlin-модули.

Карта
Основной вариант - Google Maps Compose

Для первого MVP это самый практичный выбор:

привычная карта;
хорошая работа жестов и камеры;
готовая кластеризация;
кастомные маркеры;
можно выводить фотографию внутри метки;
при уменьшении масштаба объединять фотографии в группы.

Maps Compose предоставляет отдельный Clustering composable. Для одиночных элементов и кластеров можно передавать собственный Compose-интерфейс, поэтому реально сделать круглые или квадратные превью фотографий, как в Google Photos.

Недостаток - нужен Google Cloud API key и подключённый биллинг. Тарифы Google Maps со временем меняются, поэтому перед публикацией приложения нужно установить квоты и уведомления о расходах.

Альтернатива - MapLibre

Подойдёт, если принципиально не хочется зависеть от Google:

открытый движок;
можно использовать собственные или сторонние тайлы;
больше контроля над стилем карты;
нет привязки к Google Maps Platform.

Но настройка тайлов, стилей и кластеризации будет сложнее. Для первого Android-проекта я бы начал с Google Maps, а MapLibre рассматривал после рабочего MVP. MapLibre Native поддерживает Android, а MapLibre Compose предоставляет Compose-обёртку для карт.

Как хранить данные

Оригинальные фотографии копировать в приложение не нужно. В Room достаточно хранить индекс:

@Entity
data class PhotoEntity(
    @PrimaryKey
    val mediaId: Long,

    val uri: String,
    val fileName: String?,
    val takenAt: Long?,
    val latitude: Double,
    val longitude: Double,
    val width: Int?,
    val height: Int?,
    val modifiedAt: Long
)

Изображение загружается по content:// URI через Coil. Coil поддерживает Kotlin, Coroutines и Jetpack Compose, а также занимается декодированием и кешированием изображений.

Предлагаемая архитектура
MediaStore
    ↓
PhotoScanner
    ↓
ExifInterface
    ↓
Room Database
    ↓
PhotoRepository
    ↓
ViewModel + StateFlow
    ↓
Compose UI
    ├── GalleryScreen
    ├── MapScreen
    └── PhotoDetailsScreen

Состояние карты и галереи стоит хранить во ViewModel и передавать в UI через StateFlow. Это соответствует рекомендуемой Android-архитектуре, где ViewModel преобразует данные репозитория в состояние экрана.

Как будет работать карта
При первом запуске пользователь даёт доступ к фотографиям.
Приложение получает список изображений через MediaStore.
Для каждого фото считывается EXIF.
Фото без координат остаются в обычной галерее, но не отображаются на карте.
Фото с координатами записываются в Room.
На карте отображаются только фотографии в текущей видимой области.
Близкие фотографии объединяются в кластеры.
При нажатии на кластер карта приближается.
При нажатии на одиночную метку открывается превью или экран фотографии.

Повторное индексирование можно запускать через WorkManager, поскольку он предназначен для надёжных фоновых задач, которые должны завершиться даже после закрытия приложения.

Важный момент с производительностью

Не стоит передавать на карту сразу все 20 000 фотографий.

Лучше:

хранить координаты в Room;
получать фотографии только для текущих границ карты;
использовать кластеризацию;
загружать миниатюры уменьшенного размера;
кешировать изображения;
пересчитывать набор меток после завершения перемещения карты;
не читать EXIF заново при каждом запуске.

Для первой версии достаточно обычных индексов по latitude и longitude. При очень большой медиатеке можно добавить SQLite R-Tree или отдельный геопространственный индекс.

Без сервера, регистрации и облачной синхронизации. Приложение сначала полностью работает локально с фотографиями телефона.

Итоговый выбор: Kotlin + Jetpack Compose + Google Maps Compose + Room + MediaStore. Для Android-галереи с фотографиями на карте это даст меньше проблем с разрешениями, EXIF, производительностью и системной медиатекой, чем кроссплатформенный стек.
