Техническое задание: автоматическая обработка новых фотографий
1. Контекст

Ты работаешь в существующем Android-приложении локальной фотогалереи с фотографиями на карте.

Приложение:

читает изображения через MediaStore;
сохраняет локальный индекс фотографий в Room;
извлекает дату и GPS из EXIF;
отображает фотографии через MapLibre;
определяет поездки;
сопоставляет фотографии с географическими объектами;
формирует пользовательские места;
строит тепловую карту;
не отправляет фотографии и координаты на сервер.

Не создавай новый проект и не переписывай существующие функции.

Перед началом:

Изучи текущую архитектуру проекта.
Найди код первичного сканирования MediaStore.
Найди таблицу фотографий Room.
Найди Worker, отвечающий за анализ фотографий.
Найди репозитории поездок, мест и тепловой карты.
Проверь используемый DI.
Проверь разрешения на чтение изображений.
Проверь поддержку Android 10-16.
Составь список файлов, которые нужно изменить или создать.
2. Цель

Реализовать автоматическую инкрементальную обработку новых, изменённых и удалённых фотографий.

После появления новой фотографии приложение должно:

обнаружить изменение MediaStore
-> найти только новые или изменённые записи
-> сохранить базовые данные в Room
-> прочитать EXIF
-> определить координаты
-> сопоставить фотографию с географическими объектами
-> обновить места
-> обновить затронутые поездки
-> обновить затронутые ячейки тепловой карты
-> автоматически обновить UI

Не выполнять полное сканирование всей галереи после добавления каждой фотографии.

3. Основные требования

Система должна поддерживать:

новые фотографии;
изменённые фотографии;
удалённые фотографии;
фотографии, добавленные массово;
фотографии, добавленные при закрытом приложении;
фотографии, импортированные с прошлыми датами;
изменение EXIF;
изменение даты съёмки;
добавление или удаление GPS;
перенос файла между папками;
потерю доступа к фотографии;
частичный доступ к галерее;
перезагрузку устройства;
прерывание фоновой обработки.

9. Инкрементальная синхронизация

Для поддерживаемых версий Android использовать generation MediaStore.

Алгоритм:

1. Прочитать сохранённые version и generation.
2. Получить текущую MediaStore version.
3. Если version изменилась - выполнить полную синхронизацию.
4. Зафиксировать targetGeneration.
5. Найти записи с generationModified:
   больше сохранённой generation
   и не больше targetGeneration.
6. Обработать найденные записи.
7. Сохранить изменения в Room.
8. Только после успешной транзакции сохранить targetGeneration.
9. Повторно проверить generation.
10. Если generation выросла - выполнить ещё один проход.

Не обновлять checkpoint до успешного завершения обработки всех записей.

10. Fallback для старых версий Android

Если generation API недоступен, использовать комбинацию:

_ID;
DATE_ADDED;
DATE_MODIFIED;
локальный индекс Room.

Fallback-алгоритм:

1. Получить дату последней успешной синхронизации.
2. Найти записи с DATE_ADDED или DATE_MODIFIED после checkpoint.
3. Сравнить найденные строки с Room.
4. Обработать новые и изменённые записи.
5. Периодически выполнять полную сверку ID.

Не полагаться только на имя файла.

Основной идентификатор:

volumeName + mediaStoreId
11. Worker синхронизации

Создать:

sync/PhotoLibrarySyncWorker.kt

Пример:

class PhotoLibrarySyncWorker(
    appContext: Context,
    params: WorkerParameters,
    private val synchronizer: PhotoLibrarySynchronizer
) : CoroutineWorker(appContext, params) {

    override suspend fun doWork(): Result {
        return try {
            repeat(MAX_SYNC_PASSES) {
                val result = synchronizer.synchronize()

                if (!result.hasMoreChanges) {
                    return Result.success()
                }
            }

            Result.retry()
        } catch (exception: SecurityException) {
            Result.failure()
        } catch (exception: IOException) {
            Result.retry()
        } catch (exception: CancellationException) {
            throw exception
        } catch (exception: Exception) {
            Result.retry()
        }
    }

    companion object {
        private const val MAX_SYNC_PASSES = 5
    }
}

Требования:

использовать CoroutineWorker;
корректно обрабатывать отмену;
не выполнять тяжёлую работу на main thread;
использовать внедрение зависимостей через текущий DI;
не хранить Activity или ViewModel;
не создавать новый экземпляр базы вручную, если база уже предоставляется DI.

15. Обработка новой фотографии

Для каждой новой фотографии выполнить:

1. Сохранить базовую строку MediaStore в Room.
2. Проверить доступность content URI.
3. Проверить размер файла.
4. Прочитать дату съёмки.
5. Прочитать GPS EXIF.
6. Нормализовать координаты.
7. Проверить корректность даты.
8. Сопоставить координату с географическими объектами.
9. Отфильтровать мусорные геообъекты.
10. Создать связи фотографии с местами.
11. Определить затронутый временной сегмент поездок.
12. Пересчитать затронутые поездки.
13. Обновить heatmap.
14. Пометить фотографию как COMPLETED.

Не блокировать всю синхронизацию из-за одной повреждённой фотографии.

16. Чтение EXIF

EXIF читать через contentResolver.openInputStream() или существующий способ работы с content:// URI.

Не преобразовывать URI в абсолютный файловый путь.

Обработать:

отсутствие DateTimeOriginal;
отсутствие GPS;
повреждённый EXIF;
некорректные координаты;
координаты (0, 0);
дату из будущего;
файл, который ещё не завершил запись;
временную ошибку открытия потока.

При временной ошибке:

processingState = FAILED
processingError = temporary

и повторить обработку позже.

17. Незавершённые файлы

Не обрабатывать незавершённые записи MediaStore.

Проверять:

размер файла больше нуля;
запись доступна через ContentResolver;
IS_PENDING не установлен, если колонка доступна.

Если файл временно недоступен:

не удалять его;
не помечать окончательно повреждённым;
запланировать повторную обработку.

val affectedPlaceIds =

Не пересчитывать все пользовательские места.

19. Обновление поездок

Новая фотография может:

попасть в существующую поездку;
расширить начало поездки;
расширить конец поездки;
объединить два сегмента;
создать новую поездку;
оказаться домашней фотографией;
изменить старую поездку при импорте архива.

Для каждой фотографии определить окно перерасчёта:

dateTaken - 7 дней
dateTaken + 7 дней

Это минимальный вариант.

Предпочтительный вариант:

ближайший устойчивый домашний сегмент до фотографии
+
ближайший устойчивый домашний сегмент после фотографии

Пересчитать только фотографии внутри затронутого временного окна.

Если dateTaken отсутствует, фотография не участвует в определении поездки до появления корректной даты.

20. Обновление тепловой карты

После изменения мест или поездок:

Получить старые затронутые H3-ячейки.
Получить новые затронутые H3-ячейки.
Объединить два множества.
Пересчитать только эти ячейки.
Увеличить версию данных heatmap.

Пример:

heatmapRepository.rebuildCells(
    affectedCellIds
)

Не выполнять полный пересчёт heatmap для одной фотографии.

21. Изменённые фотографии

Перед обновлением фотографии сохранить старое состояние влияния:

data class PreviousPhotoImpact(
    val oldTakenAt: Long?,
    val oldLatitude: Double?,
    val oldLongitude: Double?,
    val oldPlaceIds: Set<String>,
    val oldTripIds: Set<Long>,
    val oldHeatCellIds: Set<String>
)

После повторного чтения метаданных:

1. Сравнить старые и новые данные.
2. Удалить неактуальные связи с местами.
3. Добавить новые связи.
4. Пересчитать старые места.
5. Пересчитать новые места.
6. Пересчитать старое временное окно поездок.
7. Пересчитать новое временное окно поездок.
8. Пересчитать старые и новые heat-ячейки.

Пример:

GPS был добавлен:
-> фотография появляется на карте;
-> добавляются места;
-> может измениться поездка.

GPS был удалён:
-> удаляются связи с местами;
-> фотография исчезает с карты;
-> пересчитывается поездка.

Дата была изменена:
-> фотография переносится между днями и поездками.
22. Удалённые фотографии

Периодически сверять:

MediaStore IDs
с
Room mediaStore IDs

Алгоритм:

1. Получить все доступные ID из MediaStore.
2. Получить isAvailable=true ID из Room.
3. Найти ID, которые отсутствуют в MediaStore.
4. Пометить их isAvailable=false.
5. Удалить связи с географическими объектами.
6. Пересчитать затронутые места.
7. Пересчитать затронутые поездки.
8. Удалить вклад в heatmap.
9. Очистить локальные preview cache entries.

Не удалять строку фотографии сразу, если она нужна для восстановления связей и отладки.

Можно выполнять физическую очистку позже отдельной задачей.

23. Полная синхронизация

Полную синхронизацию выполнять:

при первом запуске;
после изменения MediaStore version;
после изменения версии алгоритма синхронизации;
после потери checkpoint;
после повреждения локального индекса;

Полная синхронизация не должна автоматически означать полный повторный анализ всех фотографий.

Для каждой строки сравнивать:

дату изменения;
размер;
processingVersion.

Повторно обрабатывать только те фотографии, которые действительно требуют обновления.

30. Транзакции

Внутри одной транзакции сохранять:

изменённые PhotoEntity;
checkpoint generation;
связи с местами;
затронутые агрегаты, если это возможно.

Checkpoint нельзя сохранять раньше данных фотографий.

При отмене или ошибке повторный запуск должен безопасно обработать тот же диапазон generation.

Все операции должны быть идемпотентными.

31. Идемпотентность

Повторный запуск одной синхронизации не должен:

создавать дубликаты фотографий;
создавать дубликаты связей с местами;
удваивать количество посещений;
создавать дубликаты поездок;
дважды увеличивать heatmap;
повторно отправлять одно уведомление.

Использовать стабильные первичные ключи и upsert.

32. Кэширование

После обновления данных инвалидировать только нужные кэши:

карта фотографий;
карточки затронутых поездок;
карточки затронутых мест;
FeatureCollection затронутого heatmap-уровня;
блок «Этот день в прошлом», если новая фотография подходит по дате.

Не очищать весь кэш приложения после каждой фотографии.

37. Ручная проверка через ADB

После реализации проверить на эмуляторе.

Создать папку:

$adb = "$env:LOCALAPPDATA\Android\Sdk\platform-tools\adb.exe"
$serial = "emulator-5556"

& $adb -s $serial shell mkdir -p `
    /sdcard/Pictures/PhotoMapTest

Добавить фотографию:

& $adb -s $serial push `
    "D:\test\fdfds.jpg" `
    "/sdcard/Pictures/PhotoMapTest/new-photo.jpg"

Запустить Media Scanner:

& $adb -s $serial shell am broadcast `
    -a android.intent.action.MEDIA_SCANNER_SCAN_FILE `
    -d "file:///sdcard/Pictures/PhotoMapTest/new-photo.jpg"

Ожидается:

фотография появляется в MediaStore;
observer планирует Worker;
запись появляется в Room;
после обработки фотография появляется в приложении.

Удаление:

& $adb -s $serial shell rm `
    /sdcard/Pictures/PhotoMapTest/new-photo.jpg

После удаления запустить синхронизацию и проверить, что запись стала недоступной.

38. Производительность

Обязательные ограничения:

не читать всю галерею после каждой фотографии;
не выполнять EXIF-анализ на main thread;
не создавать Worker на каждый файл;
не обновлять UI после каждого шага одной фотографии;
не пересчитывать все поездки;
не пересчитывать все места;
не перестраивать всю heatmap;
не декодировать полноразмерные изображения;
не хранить Bitmap в Room;
не передавать большие списки через WorkManager input data;
не передавать FeatureCollection через WorkManager;
не хранить Activity Context в Worker или репозитории.
39. Критерии приёмки

Задача считается выполненной, когда:

приложение автоматически обнаруживает новые фотографии;
новые фотографии появляются без ручного полного сканирования;
используется unique WorkManager task;
несколько MediaStore-событий объединяются в одну обработку;
используется инкрементальная синхронизация;
checkpoint сохраняется только после успешной обработки;
повторный запуск не создаёт дубликаты;
новые фотографии с GPS появляются на карте;
фотографии без GPS сохраняются без ошибки;
связанные места обновляются;
затронутые поездки пересчитываются;
heatmap обновляется инкрементально;
изменённые фотографии корректно переобрабатываются;
удалённые фотографии исчезают из активного индекса;
приложение корректно работает после перезапуска;
поддержан частичный доступ к галерее;
UI не блокируется;
добавлены unit- и integration-тесты;
проект успешно собирается;
существующая карта фотографий не сломана.
