# Branch `feature-14-trip-segmentation`

## Release Target

```text
v0.9.0
```

## Задача

Добавить в Photo Map семантическую кластеризацию поездок: приложение должно автоматически находить поездки как непрерывные временные эпизоды, когда пользователь покидает обычный базовый регион, находится в другом регионе или нескольких регионах, а затем возвращается к обычной географии.

Это не замена текущей визуальной кластеризации MapLibre. Визуальные кластеры на карте остаются отдельным слоем отображения. Новая фича должна быть доменным слоем над уже проиндексированными фотографиями с координатами и датой съемки.

## Контекст

В проекте уже есть:

* локальный индекс фотографий;
* координаты фотографий;
* дата съемки / временные поля;
* экран карты с визуальной кластеризацией;
* фильтр карты по датам;
* локальная обработка без сервера.

Использовать эти данные как источник. Не перечитывать EXIF ради построения поездок, если координаты и дата уже есть в локальной базе.

## Цель

Сформировать локальный индекс поездок, который позволит:

* выделять поездки из истории фотографий;
* отличать поездку от обычных домашних/регулярных регионов;
* поддерживать одноцентровые и многоцентровые поездки;
* хранить границы поездки по датам;
* хранить центр, радиус, уверенность и список фотографий поездки;
* показывать найденные поездки на отдельной карте поездок;
* открывать карту поездок с основной карты по FAB `Поездки`;
* открывать отдельную страницу поездки по нажатию на кластер поездки;
* показывать на странице поездки хронологическую линию маршрута по карте и сетку фотографий поездки.

## Ключевая модель предметной области

Разделить понятия:

* `Trip` - непрерывный временной интервал поездки;
* `TripDestination` - город, район или место внутри поездки;
* `PhotoMapCluster` - визуальная группа фотографий на текущем масштабе карты;
* `TransitPhoto` - фотография в пути, которая входит в интервал поездки, но не обязана формировать destination.

Не смешивать поездки с текущими кластерами маркеров карты.

## MVP-алгоритм

Для первой реализации использовать эвристический алгоритм без сложного ML:

1. Взять все фотографии с валидными координатами и датой съемки.
2. Отсортировать по времени.
3. Объединить близкие фотографии в фотосессии, чтобы серия из десятков снимков за несколько минут не имела вес в десятки раз больше одной сессии.
4. Определить базовые регионы пользователя по устойчивым повторяющимся местам.
5. Построить дневную последовательность географических наблюдений.
6. Найти непрерывные away-сегменты, удаленные от базового региона.
7. Проверить сегменты правилами поездки: удаленность от базы, доля дней/сессий внутри региона, контраст границ до и после, длительность, число активных дней.
8. Для подтвержденных поездок определить центр, радиус, тип и confidence.
9. Для фотографий внутри поездки определить destinations простым пространственным clustering/stay-region подходом.
10. Сохранить результат локально.

## Рекомендуемые эвристики MVP

### Фотосессии

Объединять соседние фотографии в одну сессию, если:

```text
time gap <= 3 часа
distance <= 20 км
```

Вес поездки считать по активным дням и фотосессиям, а не по количеству фотографий.

### Базовые регионы

Базовый регион - место, которое стабильно встречается в большом числе разных дней и месяцев. Не выбирать базу только по количеству фотографий.

Стартовая формула веса:

```text
baseScore = uniqueVisitedDays + activeMonths * monthWeight
```

Поддержать несколько базовых регионов, чтобы регулярные возвращения во второй город не определялись как поездки.

### Дневное наблюдение

Добавить доменную модель уровня алгоритма:

```kotlin
data class DailyLocationObservation(
    val date: LocalDate,
    val centerLatitude: Double?,
    val centerLongitude: Double?,
    val distanceToNearestBaseKm: Double?,
    val dispersionKm: Double?,
    val dominantRegionId: String?,
    val sessionCount: Int
)
```

Центр дня считать медоидом или другой устойчивой оценкой, а не обычным средним, если это возможно без чрезмерного усложнения.

### Кандидат поездки

Сегмент может считаться поездкой, если стартовые условия выполняются достаточно уверенно:

```text
distanceFromBase >= 50 км
duringShare >= 0.65
boundaryContrast >= 0.70
activeDays >= 1
duration <= 60 дней
```

Эти значения вынести в константы или настройки алгоритма, не хардкодить по месту использования.

### Confidence

Для каждой поездки считать итоговую уверенность:

```text
tripConfidence =
    0.35 * insideDuring
    + 0.25 * min(outsideBefore, outsideAfter)
    + 0.20 * distanceFromBaseScore
    + 0.10 * temporalContinuity
    + 0.10 * activeDayCoverage
```

Если нет явного подтверждения возвращения в базовый регион, снижать confidence, а не обязательно удалять поездку.

### Радиус поездки

Для одноцентровой поездки:

```text
center = медоид фотосессий
radius = clamp(Q80(distanceToCenter), 2 км, 50 км)
```

Не использовать максимальное расстояние как радиус, чтобы случайная фотография из самолета или трассы не ломала область поездки.

### Типы поездок

Поддержать минимум:

```kotlin
enum class TripType {
    SINGLE_DESTINATION_TRIP,
    MULTI_DESTINATION_TRIP,
    UNKNOWN
}
```

Правило:

* если один удаленный пространственный кластер покрывает `>= 65%` сессий, это `SINGLE_DESTINATION_TRIP`;
* если несколько удаленных от базы кластеров идут подряд во времени, это `MULTI_DESTINATION_TRIP`;
* если данных недостаточно, тип `UNKNOWN`, но поездку можно сохранить с низким confidence.

### Дни без фотографий

День без фотографий не означает возвращение домой.

Правила:

* пробел 1-2 дня внутри устойчивого away-сегмента не разрывает поездку;
* пробел больше 2 дней помечать как unknown-участок;
* поездка продолжается, если после пробела фотографии снова появляются в том же away-регионе;
* окончание поездки лучше подтверждать фотографиями из базового региона.

## UI

Добавить отдельный пользовательский сценарий для поездок:

* на основной карте добавить FAB `Поездки`;
* при нажатии на FAB открывать новый экран карты поездок;
* карта поездок должна быть отдельным экраном, а не режимом основной карты;
* карта поездок должна визуально отличаться от основной карты темной темой;
* для карты поездок использовать отдельный MapLibre style URL `TRIP_MAP_STYLE_URL`;
* значение по умолчанию для `TRIP_MAP_STYLE_URL` - `https://tiles.openfreemap.org/styles/dark`;
* на карте поездок каждая найденная поездка отображается как один крупный кластер;
* кластер поездки выглядит как миниатюра одной фотографии с бейджем `+N`, где `N` - число фотографий в поездке;
* при нажатии на кластер поездки открывать отдельную страницу поездки внутри приложения;
* страница поездки должна показывать MapLibre-карту с линией маршрута поездки;
* заголовок страницы поездки должен показывать ближайший населенный пункт или несколько пунктов маршрута, если их удалось определить через платформенный geocoding API;
* пока название/локация поездки загружается, в верхней панели показывать skeleton вместо временного fallback-текста;
* дата поездки должна показываться под локацией; технический id поездки в шапке не показывать;
* не добавлять внешний reverse-geocoding SDK, API key или прямую отправку координат в сторонний сервис; если geocoder недоступен, показывать только статистику поездки;
* линия маршрута строится по фотографиям поездки в хронологическом порядке;
* начало линии - первая хронологическая фотография, конец - последняя хронологическая фотография;
* линия должна быть сглаженной, цвет линии должен идти градиентом от начала к концу, точки фотографий должны быть видны на линии;
* точки фотографий на маршруте должны показывать миниатюры, быть кликабельными и скроллить нижнюю галерею к выбранной фотографии;
* под картой страницы поездки должна быть `LazyVerticalGrid`-галерея фотографий этой поездки;
* клик по фотографии в нижней галерее должен открывать снимок в системной/дефолтной галерее Android через read-only `ACTION_VIEW`;
* на странице поездки добавить такой же минимальный правый scrubber, но по фотографиям внутри поездки;
* scrubber фотографий должен выбирать фото по хронологии, центрировать карту маршрута на выбранной точке и скроллить нижний grid к выбранному снимку;
* на странице поездки добавить действие центрирования маршрута, которое сбрасывает zoom/центр карты на всю поездку;
* при возврате со страницы поездки на карту поездок сохранять zoom/центр выбранной поездки, не отдалять карту обратно ко всем поездкам;
* на карте поездок добавить вертикальный scrubber годов у правого края экрана;
* scrubber поездок должен выглядеть как длинный timeline, допускается выход timeline за границы экрана;
* активная точка timeline должна удерживаться около видимой зоны, а выбор должен снапиться к ближайшей точке поездки, включая соседние точки до/после;
* активная поездка в timeline должна подсвечиваться акцентным цветом и рамкой, соседние точки остаются приглушенными;
* если поездка одна, timeline должен отображаться как одна точка без вертикальной линии;
* scrubber должен быть минимальным: без рамки и фоновой карточки, только примерно 25% прозрачные отметки;
* сбоку от точек scrubber показывать подписи поездок с прозрачностью около 33%;
* при tap/drag по scrubber выбирать ближайшую поездку по дате и анимированно перемещать карту к этой поездке;
* если поездки еще не рассчитаны или не найдены, показывать понятное пустое состояние;
* действие ручного пересчета поездок можно разместить на карте поездок или в настройках, по текущей архитектуре проще выбрать один вход.

Не открывать поездку через фильтр дат на основной карте как основной UX. Основной пользовательский вход для новой фичи - FAB `Поездки` и отдельная карта поездок.

### Карта поездок

Карта поездок должна переиспользовать MapLibre/AndroidView-подход проекта, но иметь собственный `UiState`, ViewModel и слой отображения.

Требования:

* использовать темный style URL `TRIP_MAP_STYLE_URL`, переопределяемый через `local.properties`;
* применять темную Material-обвязку, темные панели и акцентные цвета поверх карты поездок;
* не ломать основной `MAP_STYLE_URL`;
* центрировать карту так, чтобы были видны все поездки;
* не показывать обычные фото-маркеры и текущие visual photo clusters на карте поездок;
* отображать только агрегаты поездок;
* миниатюру поездки брать из первой/обложечной фотографии поездки;
* бейдж показывать как `+327`, где число равно `photoCount`;
* если у поездки нет доступной миниатюры, использовать аккуратный placeholder;
* кластеры поездок должны оставаться кликабельными и иметь стабильные размеры при изменении масштаба;
* карта должна корректно работать после поворота экрана.

Минимальная модель для UI:

```kotlin
data class TripMapMarker(
    val tripId: Long,
    val coverPhotoId: Long?,
    val coverPhotoUri: String?,
    val latitude: Double,
    val longitude: Double,
    val photoCount: Int,
    val startDate: Long,
    val endDate: Long,
    val confidence: Double,
    val type: TripType
)
```

Добавить маршрут Navigation Compose, например:

```text
trips
trip/{tripId}
```

`trips` - отдельная карта поездок. `trip/{tripId}` - отдельная страница поездки с картой маршрута и grid-галереей фотографий поездки.

Навигацию новых и существующих экранов вести через Navigation Compose (`NavHost`, `composable`, `rememberNavController`), а не через ручной стек экранов.

Не добавлять отдельный пункт нижней навигации для поездок в первой версии, если есть FAB на основной карте.

## Фоновая и ручная пересегментация

Пересегментация поездок должна запускаться:

* вручную из UI;
* после завершения полного сканирования фотографий, если это вписывается в текущий pipeline;
* без параллельных запусков.

Алгоритм должен работать на `Dispatchers.Default` или `Dispatchers.IO` по смыслу операции и не блокировать главный поток.

Для больших медиатек не передавать огромные mutable-списки в Compose. В UI отдавать только готовые `UiState`.

## Безопасность и приватность

Запрещено:

* отправлять фотографии, координаты, EXIF, URI, mediaId или статистику поездок в сеть;
* добавлять аналитику;
* добавлять reverse geocoding через внешний API;
* изменять, удалять, перемещать или перезаписывать оригинальные фотографии;
* записывать EXIF;
* добавлять `WRITE_EXTERNAL_STORAGE` или `MANAGE_EXTERNAL_STORAGE`.

Все поездки и промежуточные данные должны храниться локально.

## Что не входит в эту ветку

* HMM/HSMM;
* Bayesian Online Change Point Detection;
* полноценный PELT с внешними библиотеками, если он заметно усложняет первую версию;
* серверная обработка;
* синхронизация с Google Photos;
* reverse geocoding и названия городов из сети;
* ручное редактирование границ поездки, если MVP списка и открытия на карте еще не готов;
* изменение текущей логики визуальных кластеров MapLibre без необходимости.

## Тестовые сценарии для реализации

Добавить unit-тесты на алгоритмическую часть:

* серия фотографий в базовом регионе не создает поездку;
* базовый регион -> другой город -> базовый регион создает одну поездку;
* повторная поездка в тот же город через месяц создает отдельную поездку;
* маршрут через несколько городов создает `MULTI_DESTINATION_TRIP`, а не несколько несвязанных поездок;
* день без фотографий внутри away-сегмента не разрывает поездку;
* одна случайная фотография далеко от базы не создает уверенную поездку;
* 500 фотографий за один день не перевешивают 10 активных дней по одной фотографии;
* радиус поездки устойчив к выбросам;
* поездка без подтвержденного возвращения получает сниженный confidence;
* фотографии без даты или координат пропускаются и не ломают сегментацию.

Добавить ViewModel-тесты для списка поездок и ручного запуска пересегментации, если такие ViewModel появятся.

## Ручные проверки после реализации

После реализации вручную проверить:

* приложение открывается без падений после миграции базы;
* список поездок показывает пустое состояние на медиатеке без GPS-фотографий;
* ручной пересчет не блокирует UI;
* FAB `Поездки` на основной карте открывает отдельную темную карту поездок;
* кластеры поездок на карте показывают миниатюру и бейдж `+N`;
* нажатие на кластер поездки открывает страницу поездки с картой маршрута;
* линия маршрута идет по карте от первой хронологической фотографии к последней;
* линия маршрута сглажена и сохраняет градиент начала/конца;
* заголовок страницы поездки показывает ближайшие населенные пункты маршрута вместо id, если geocoder доступен;
* дата поездки показана второй строкой под локацией;
* миниатюры на точках маршрута кликабельны и переводят пользователя к фотографии в нижнем grid;
* клик по фотографии в grid открывает ее в дефолтной галерее Android;
* правый scrubber на странице поездки позволяет перемещаться между фотографиями внутри поездки;
* кнопка центрирования на странице поездки сбрасывает zoom на всю поездку;
* возврат со страницы поездки сохраняет zoom/центр выбранной поездки на карте поездок;
* под картой поездки отображается grid с фотографиями выбранной поездки;
* текущие маркеры и кластеры карты продолжают работать;
* галерея и просмотр фото не ломаются;
* при отсутствии разрешений экран поездок не падает и показывает понятное состояние.

## Acceptance Criteria

Задача считается выполненной, когда:

* добавлен локальный доменный слой поездок;
* поездки строятся из уже проиндексированных фотографий с координатами и датой;
* текущая визуальная кластеризация карты не используется как источник истины для поездок;
* результаты сохраняются в локальном индексе приложения;
* на основной карте есть FAB `Поездки`;
* есть отдельный экран темной карты поездок;
* найденные поездки отображаются на карте поездок как кластеры-миниатюры с бейджем `+N`;
* нажатие на кластер поездки открывает отдельную страницу `trip/{tripId}`;
* страница `trip/{tripId}` показывает даты поездки, населенные пункты при доступном geocoder, сглаженную тепловую линию маршрута по карте и grid фотографий поездки;
* миниатюры на точках маршрута кликабельны и скроллят grid к выбранному фото;
* кнопка центрирования на странице поездки сбрасывает zoom/центр карты маршрута на всю поездку;
* правый timeline поездок снапится к ближайшим точкам поездок, может выходить за границы экрана и показывает подписи поездок с прозрачностью около 33%;
* алгоритм поддерживает одноцентровые и многоцентровые поездки;
* дни без фотографий не разрывают поездку автоматически;
* случайные выбросы снижают confidence или попадают в transit/noise, но не ломают радиус поездки;
* добавлены focused unit-тесты алгоритма;
* README/CHANGELOG/release notes обновлены под `v0.9.0`;
* не добавлены сетевые API, аналитика, write/manage-разрешения или операции изменения оригинальных фото.
